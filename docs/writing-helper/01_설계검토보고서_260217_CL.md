## A. 치명적 문제 (구현 전 반드시 해결)

### A-1. 다음자(多音字) 처리 완전 누락

**관련**: 08 엔진 설계 — `toneValidator`

현재 `pingshui_yun.json`의 `char_map`은 한 글자당 하나의 성조만 매핑:

```json
"占": {"tone": "平", "rhyme_key": "下平十四鹽"}
```

그러나 `占`은 평성(zhān, 점치다)과 측성(zhàn, 차지하다) 두 가지 읽기가 있음. 이런 다음자가 수백 개:

- `中`: zhōng(평) / zhòng(측)
- `行`: háng(평) / xíng(측)
- `長`: cháng(평) / zhǎng(측)
- `勝`: shēng(평) / shèng(측)

**현재 char_map이 1:1 매핑이므로 평측 판정이 부정확할 수 있음.** 08 문서에서 이 문제를 전혀 언급하지 않음.

**해결 방향 제안**:
- char_map을 배열 구조로 확장: `"占": [{"tone":"平","rhyme_key":"下平十四鹽"}, {"tone":"仄","rhyme_key":"去二十九艷"}]`
- 또는 다음자 별도 테이블 + 문맥 기반 추정 로직
- `generate_pinyin_pingze.py`의 `VARIANT_MAP`, `DIRECT_TONE_MAP`에 이미 일부 예외 처리가 있으므로 이를 확장하는 방향도 가능
-> 제안내용 컨펌
---

### A-2. 입성(入聲) 처리 미설계

**관련**: 08 엔진 설계 — `toneValidator`, `illnessDetector`

우리 운서에 **입성 17운부**(820자 이상)가 있음. 입성은 측성에 속하는데, 현대 중국어(보통화)에서는 1~4성으로 흩어져 사라짐. 한국어 한자음으로도 구분이 쉽지 않음.

현재 char_map에 입성 글자는 "仄"으로 들어있긴 하지만:
- 사용자에게 "이 글자는 입성이므로 측성"이라는 **별도 표시**가 없으면 학습 효과 감소
- 입성 특유의 운율 느낌을 설명하는 코칭도 필요

**해결 방향 제안**: char_map에 `"toneDetail": "入"` 같은 상세 성조 필드 추가, UI에서 입성 글자 시각적 구분 표시
-> 제안내용 컨펌

---

### A-3. 한자 입력 방법 완전 누락

**관련**: 09 UI 기획 — Step 5, Step 6 전체

**가장 큰 UX 구멍.** 한국 사용자가 한시를 쓸 때 한자를 어떻게 입력하는지 전혀 언급 없음:

- 한글 IME → 한자 변환? (일반적이나 시어 찾기 어려움)
- 병음 검색? (중국어 모르는 사용자 많음)
- 운서 DB에서 뜻/부수/획수 검색?
- 한글 뜻 입력 → 한자 후보 선택?

**이 부분이 해결 안 되면 Step 5(압운 자리 채우기), Step 6(본문 작성)의 핵심 창작 경험 자체가 불가능.** 모바일에서는 더 심각.

**해결 방향 제안**:
- 1순위: 한글 IME 한자변환 기본 지원 (OS 기능 활용)
- 2순위: 자체 한자 검색 UI — 한글 뜻 입력 → 후보 한자 표시 (CC-CEDICT 한중 사전 활용)
- 3순위: 운서 내 브라우징 — 운부별 글자 목록 스크롤/검색
- Step 5(압운)에서는 운부가 이미 정해져 있으므로, 해당 운부 글자 목록에서 고르는 UI가 자연스러움

## B. 중요한 누락 (설계 보완 필요)

### B-1. 배율(排律) — 예시 데이터 없음

**관련**: 07, 08, 09 전체 — 형식 선택에 배율 포함

3개 문서 모두 배율을 지원 형식으로 명시. 그러나 **우리 DB(poems.full.json)에 배율이 한 편도 없음:**

```
五言律詩: 80편  |  七言律詩: 53편
五言絶句: 29편  |  七言絶句: 51편
五言古詩: 35편  |  七言古詩: 28편  ← 고체시 (근체시 아님)
五言樂府: 18편  |  七言樂府: 26편  ← 고체시 (근체시 아님)
```

"학습 영역"에서 배율 탭을 만들어도 보여줄 대표 작품이 없음. 배율은 구조도 복잡(가변 련수).

**제안**: MVP에서 배율 제외, 또는 외부 예시 작품 별도 수급 필요. → **형 결정 필요**
-> 배율 제외가 맞음. 배율은 어려워서 잘 쓰지도 않고 우리 AI로 검증이 가능할지도 의문. 배율 예시 작품 몇개 소개하는 정도로 일단 제외하는게 맞을듯. 
---

### B-2. 기존 데이터 자산과의 연결 부재

**관련**: 07, 08, 09 전체

3개 문서 어디에도 우리 프로젝트의 기존 데이터를 참조하지 않음:

| 우리가 가진 것 | 파일 | 활용 가능성 |
|--------------|------|-----------|
| 평수운 운서 DB | `public/index/pingshui_yun.json` | 106운부, 8550자 char_map → 엔진의 운서/평측 DB로 즉시 사용 가능 |
| 시 전체 데이터 | `public/index/poems.full.json` | 320편 (근체시 213편) → 학습 예시 + pingze 데이터 보유 |
| 평측 생성 스크립트 | `scripts/generate_pinyin_pingze.py` | VARIANT_MAP(38자), DIRECT_TONE_MAP(22자) → 다음자 처리 기초 자료 |
| 운서 빌드 스크립트 | `scripts/build_pingshui_db.js` | 데이터 파이프라인 이미 존재 |

**지훈이가 기존 자산을 확인하지 않고 설계한 것으로 보임.** 이 위에 쌓으면 공수가 크게 줄어듦.
->우리 리소스를 쓰는건 맞음, 다만 우리 평측이 평수운에 근거한게 아니라 현대어 발음으로 엔진으로 넣은거라 과연 저걸 평측 데이터로 삼는게 맞을지는 의문. 역으로 평측추정모델을 우리가 개발한다면 그거에 근거해서 우리 데이터를 수정하는게 맞을듯

### B-3. form.type 네이밍 불일치

**관련**: 08 데이터 스키마 1.1절

08 문서: `"type": "qijue|wujue|qilu|wulu|pailv"` (로마자)
우리 DB: `"category": "七言絶句|五言絶句|七言律詩|五言律詩"` (한자)

매핑 테이블이 필요하거나, 우리 DB 기준으로 통일해야 함.

---

### B-4. 고체시(古體詩) 범위 미정의

**관련**: 07 학습 영역

우리 DB의 古詩 63편 + 樂府 44편 = 107편은 **근체시가 아니라 고체시**. "근체시 작성 도우미"에서:
- 학습 예시로 아예 제외?
- "비교 참고용"으로 별도 탭?

범위 정의 필요.
-> 이건 해결이 쉬워, 처음에 타입 고를때 고체시나 악부를 고르면 한자입력 도우미나 추천글자 도우미등에서 근체시 룰을 제외하면 됨
---

### B-5. rhyme.selectedRhymes 키 설계 오류

**관련**: 08 데이터 스키마 1.1절

```json
"selectedRhymes": { "2": "", "4": "", "6": "", "8": "" }
```

- **절구(4구)**: "6", "8" 키가 불필요
- **수구입운(1구 입운)**: "1" 키가 필요한데 없음
- 시체에 따라 동적으로 처리해야 함
-> 100% 맞는말

## C. 오류 및 미확인 사항

### C-1. 병(病) 탐지 범위 불완전

**관련**: 08 `illnessDetector`

08 문서가 언급한 병: **고평(孤平), 삼평미(三平尾)/삼측미(三仄尾)**

빠진 중요한 병들:
- **봉요(蜂腰)**: 2·4자 동성
- **학슬(鶴膝)**: 5자/1자 동율
- **대운(大韻)**: 앞 구에 운자와 같은 운부 글자 사용
- **소운(小韻)**: 연속 구에 같은 운부 글자

"단계적 확장"이라 했지만 **최소 어디까지 지원하고 어디부터 확장인지** 리스트업이 없음.
->위에거는 내가 언급한것만 넣은것으로 추정됨, 니가 언급한 아래거도 포함
---

### C-2. 대우 엔진의 "기초 품사 힌트" — 사전 미지정

**관련**: 08 `coupletChecker`, 대우 하이브리드 로직

"사전 기반 품사 힌트"라고 했는데 **어떤 사전?** CC-CEDICT? UniHan? 자체 구축?
공수 차이가 크므로 사전 선정이 먼저 필요.
-> 요건 나랑 상의 후 결정
---

### C-3. AI 모델/API 미지정

**관련**: 08 AI 모듈 설계 전체

AI 출력을 "항상 JSON 반환"이라 했지만:
- OpenAI GPT? Claude? Gemini?
- JSON 모드 안정성은 모델마다 다름
- 비용 추정도 모델에 따라 10배 차이 남
->이것도 내가 모르는 분야니 상의후 결정
---

### C-4. "검사 강도 프리셋" 상세 미정

**관련**: 07 `8. 내가 추가 제안하는 포인트` 1번

"초급/중급/고급" 프리셋 제안은 좋으나, 각 레벨에서 **어떤 규칙을 켜고 끄는지** 구체적 정의 없음.

예시 제안:
- 초급: 자수/구수 + 압운만 검사
- 중급: + 기본 평측(정격 위반) + 고평/삼평미
- 고급: + 전체 병 탐지 + 요구 판정 + 대우 구조
->이건 수익화 모델과 관련이 있으니 역시 상의 후 결정

## E. 잘된 점 (유지해야 할 설계)

1. **엔진/AI 분리 원칙** — "판정은 엔진, 코칭은 AI" 구조가 명확
2. **JSON 기반 상태 관리** — 재현 가능하고 디버깅 용이
3. **두 가지 모드 전략** — 자유작성(숙련자) + 실시간가이드(초보자)
4. **AI 과잉개입 방지** — 자동 적용 금지, 후보 제시 원칙
5. **요체(拗救) 처리 설계** — 빨간불→요체 전환 UX 아이디어 좋음
6. **대우 하이브리드 접근** — 엔진(구조) + AI(의미) 역할 분리
7. **비용 통제 전략** — 이벤트 기반 호출 + 캐시
8. **단계별 로드맵** — v1(MVP) → v2 → v3 점진적 확장

## G. 한자 입력 UX (A-3) — 해결됨

> **형 의견**: "이게 우리 서비스 차별점이 되어야 한다"

### 문제 인식 (형 경험 기반)

- 한시/한문 공부 시 **손으로 쓰는 건 문제없지만**, 디지털 환경(한글 키보드)에서 한자를 입력하는 것이 가장 까다로운 부분
- 현재 한글 키보드의 한자 변환 방식은 한시 창작에 매우 비효율적
- 이 문제를 잘 해결하면 **서비스의 핵심 차별점**이 됨

### 확정된 입력 방식 (2단계)

**1) 압운자 입력 — 운목 브라우징**
- 원고지 하단에 **106운목 테이블** 표시
- 운목 클릭 → 아코디언으로 소속 한자 전개
- 한자 선택 → 아코디언 닫히며 원고지에 자동 입력
- 운목 글자에 훈음은 **항상 표시** (모바일 대비, 마우스오버 불가 환경 고려)

**2) 비압운자(일반 시어) 입력 — 한글 독음 기반 자체 IME**
- 한글 독음 입력 → 한자 후보 패널 표시 (자체 구현)
- **후보 정렬**: 시어 빈도순 (poems.full.json 320편 기반 통계) — 한국어 OS IME처럼 "절 가(家)"가 먼저 나오는 게 아니라, "아름다울 가(佳)", "옳을 가(嘉)" 등 시에서 자주 쓰이는 글자 우선
- **힌트 버튼** 추가: AI가 시상 전개 노트를 참고해 10~20자 추천 → **회원 레벨에 따라 노출/비노출** (유료화 전략 연동)
- 숙련자는 OS 한자 변환도 병행 가능

### 차별점 요약

| 기존 한자 입력 | 우리 서비스 |
|--------------|-----------|
| OS 한글 IME → 일상 한자 순서 | **시어 빈도순** 정렬 |
| 운서 연동 없음 | **운목 → 아코디언** 직접 선택 |
| 문맥 무시 | **AI 힌트** (시상 기반 추천) |
| 한 가지 입력 방식 | 운목 선택 + 독음 검색 + AI 힌트 **3중 경로** |

### 미구현 세부사항 (구현 단계에서 결정)

- 후보 패널 UI 디자인 (드롭다운? 하단 패널?)
- 한자 후보에 훈음/평측 표시 여부
- 시어 빈도 DB 구축 방법 (poems.full.json 파싱 스크립트)
- 모바일은 **PC 완성 후 별도 앱으로** (MVP 범위 외)

## H. 14번 문서 리뷰: 앱 구동 시뮬레이션 5단계

> 검토 대상: `docs/FromJin/14_한시작성도우미_UI아이디어.md`

### 5단계 플로우 — 07~09 문서 대비 개선점

형이 정리한 5단계가 기존 07/08/09의 7단계 플로우보다 **사용자 관점에서 훨씬 직관적**:

| 형의 5단계 | 07~09 문서 대응 | 핵심 차이 |
|-----------|---------------|----------|
| 1. 영감 포착 (한줄 → 주제 → **한국어 초고**) | Step 1~2 | **한자 모르고도 시작 가능** — 진입장벽 대폭 하강 |
| 2. 시상 전개 (시체 선택 → **원고지 등장** → 기승전결) | Step 3~4 | **원고지 UI** 신규 개념 도입 |
| 3. 압운/운자 선택 (운목 테이블 → 아코디언) | Step 5 일부 | 구체적 UI 플로우 명시 |
| 4. 초고 작성 (**Path A**: 규칙 먼저 / **Path B**: 자유 먼저) | Step 5~6 | 두 가지 경로 분기 구체화 |
| 5. AI 퇴고 (번역→검사→수정→완성→**시화**) | Step 7 | 시화 만들기 기능 추가 |

### "원고지" UI 컨셉 — 핵심 신규 설계 요소

시체가 결정되면 등장하는 격자형 입력 인터페이스:
- **외형**: 한지 느낌 바탕 + 먹선 + 격자무늬
- **각 칸 구성**: 한자 입력 영역 + 하단 표시지(훈음, 평측)
- **신호등 시스템**: 칸 배경색 or 글자색 3가지
  - **초록**: 평측/격률 정확
  - **빨강**: 평측/격률 오류
  - **노랑**: 대우/요체 등 사후 검토 필요 항목
- 평측 표시: 글자 옆 점 방식(전통 방식)도 검토 대상

> **형 결정**: 초보/고급 구분 없이 통일 — "초보는 자기가 뭘 봐도 되는지 모르니까" 나눠봤자 위험

### Path A/B 분기 — 07문서 "모드 A/B" 고도화

| | Path A (규칙 우선) | Path B (자유 우선) |
|---|---|---|
| **운자 선택 시점** | 3단계에서 모두 선택 | 4단계에서 필요시 선택 |
| **신호등 발동** | 실시간 | 완성 후 일괄 (4단계에서는 훈음만) |
| **대상** | 격률 학습 중인 사용자 | 감각 우선 / 숙련 창작자 |
| **압운 처리** | 3단계에서 완료 | 4단계에서도 선택 가능 (선택사항) |

### 5단계 AI 퇴고 — 주요 포인트

1. **번역 확인 우선**: 한자로 작성한 시의 의미가 의도와 맞는지 먼저 검증 (한국인 특성 반영)
2. **격률 검사**: Path A는 이미 완료, Path B는 여기서 일괄
3. **수정 ↔ 재검수 반복**: AI 코칭 (회원 레벨별 검수 횟수 제한)
4. **완성 후**: 서예폰트 표시 → 개인 시첩 저장 → 공유 → 시화 만들기

### 시화(詩畵) 만들기 — 신규 기능

- 완성된 시에 어울리는 배경 이미지 선택/생성
- 프리셋 배경 vs AI 실시간 생성 (회원 레벨별 차등)
- 시 + 배경 합성 → 레이아웃 조정 → 저장/공유

> 시화 기능은 MVP 이후 검토 대상이나, 유료화 포인트로 잠재력 높음
