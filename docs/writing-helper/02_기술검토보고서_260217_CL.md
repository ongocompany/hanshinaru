## 1. 개요

설계검토 보고서에서 확정된 11건의 결정사항과 보류 5건을 기술 관점에서 검토한다. 특히:

- **엔진(규칙 기반) vs AI(LLM) 역할의 구체적 분류**
- **AI 모델 비교 및 추천**
- **한글-한자 IME 구현 난이도 및 아키텍처**
- **각 구성요소의 기술 난이도 평가**

## 2. 엔진 vs AI — 상세 역할 분류

설계 원칙: **"판정은 엔진, 코칭은 AI"**

### 2-1. 엔진 (규칙 기반, 결정적 로직)

서버 없이 **브라우저에서 즉시 실행** 가능한 항목들. 응답 지연 0ms, 비용 0원.

| 기능 | 입력 | 출력 | 데이터 의존성 | 비고 |
|------|------|------|-------------|------|
| **평측 판정** | 한자 1자 | 平/仄/? | char_map (8,550자) | A-1 다음자 배열 확장 후 사용자 선택 UI 필요 |
| **입성 표시** | 한자 1자 | toneDetail: "入" 여부 | char_map 확장 필드 | A-2 확정 |
| **압운 검사** | 구 끝자 + 선택 운부 | 일치/불일치 | char_map.rhyme_key | 운부 키 매칭만 하면 됨 |
| **자수/구수 검사** | 전체 시 | pass/fail | 시체별 규칙 테이블 | 5언=5자x4구, 7언=7자x4구 등 |
| **고평(孤平) 탐지** | 한 구의 평측열 | 경고 여부 | 평측열 패턴 매칭 | 정규식으로 구현 가능 |
| **삼평미/삼측미 탐지** | 한 구 끝 3자 | 경고 여부 | 평측열 | 끝 3자 평측만 확인 |
| **봉요(蜂腰) 탐지** | 한 구의 2·4자 | 경고 여부 | 평측열 | 2자·4자 동성 여부 |
| **학슬(鶴膝) 탐지** | 연속 두 구 | 경고 여부 | 평측열 | 5자(또는 1자) 동율 여부 |
| **대운/소운 탐지** | 연속 구들 | 경고 여부 | char_map.rhyme_key | 같은 운부 글자 검출 |
| **요체(拗救) 판정** | 위반 위치 + 구 구조 | 요체 가능 여부 | 요구 패턴 테이블 | 패턴 수 제한적 (10~15개) |
| **대우 구조 검사** | 두 구의 평측열 | 대칭 여부 | 평측열 | 위치별 평측 반전 확인 |
| **신호등 판정** | 위 검사 결과 종합 | 초록/빨강/노랑 | - | 빨강=평측오류, 노랑=대우/요체 등 |
| **rhyme 키 동적 생성** | 시체 + 기식(평/측) | 압운 위치 배열 | 시체별 규칙 | B-5 확정 |
| **한자 후보 정렬** | 한글 독음 | 빈도순 후보 리스트 | 독음→한자 DB + 빈도 DB | IME 핵심 |

> **엔진 구현 분량**: JavaScript 함수 15~20개, 총 약 800~1200줄 추정
> **핵심 데이터**: char_map(확장 후) + 요구 패턴 테이블 + 시어 빈도 DB

### 2-2. AI (LLM 호출, 비결정적)

**서버 필요, API 비용 발생, 응답 1~5초.** 이벤트 기반 호출 + 캐시로 비용 통제.

| 기능 | 호출 시점 | 입력 | 출력 | 난이도 | 비고 |
|------|----------|------|------|--------|------|
| **시체 추천** | 1단계: 한국어 초고 완성 후 | 한국어 시 텍스트 | 추천 시체 + 근거 | 낮음 | 프롬프트 엔지니어링만으로 가능 |
| **시상 전개 코칭** | 2단계: 기승전결 작성 시 | 한국어 시상 메모 | 기승전결 가이드 | 낮음 | 정형화된 패턴 조언 |
| **시어 추천 (힌트)** | 4단계: 힌트 버튼 클릭 | 현재 시 상태 + 위치 + 평측 조건 | 10~20자 후보 | **높음** | 고전 한시 어휘 + 평측 조건 동시 충족 필요 |
| **번역 확인** | 5단계: 퇴고 시작 | 완성된 한시 4~8구 | 한국어 번역 | 중간 | 고전 한문 → 한국어 번역 품질이 핵심 |
| **대우 의미 검사** | 5단계: 격률 검사 | 대구 2줄 | 의미적 대장 여부 + 설명 | **높음** | 엔진(구조) 결과 위에 의미 분석 추가 |
| **퇴고 코칭** | 5단계: 수정 후 | 전체 시 + 엔진 검사 결과 | 개선 제안 | **높음** | 가장 복잡한 AI 태스크 |

### 2-3. 하이브리드 (엔진 + AI 협업)

| 기능 | 엔진 역할 | AI 역할 |
|------|----------|---------|
| **대우(對仗) 전체 검사** | 구조적 대칭(평측 반전) 확인 → 초록/빨강 | 의미적 대장(산↔수, 풍↔월) 확인 → 코칭 |
| **요체(拗救) 안내** | 위반 위치 탐지 + 구제 가능 여부 판정 | 구제 방법 한국어 설명 + 추천 글자 |
| **전체 퇴고** | 모든 병/격률 검사 결과 JSON 생성 | JSON 기반으로 자연어 코칭 생성 |

## 3. AI 모델 비교 및 추천

### 3-1. 우리 서비스의 AI 요구사항

1. **고전 한문(文言文) 이해도** — 평측, 격률, 운서, 대장 등 전문 지식
2. **한국어 코칭 품질** — 사용자 대화는 전부 한국어
3. **JSON 구조화 출력** — 엔진과 연동하려면 정확한 JSON 필수
4. **비용 효율** — 소비자 서비스, 세션당 5~10회 호출 예상
5. **응답 속도** — 시어 추천은 2초 이내 바람직

- **한국어**: 다국어 지원 공식, 품질 양호 (GPT-4o보다는 약간 뒤지나 실용 수준)

## 4. 한글-한자 IME 구현 분석

### 4-1. 개요

우리 서비스의 **핵심 차별점**. 한글 독음을 입력하면 시어 빈도순으로 한자 후보를 보여주는 자체 입력기.

### 4-2. 필요 데이터

**1) 독음→한자 매핑 DB**

| 데이터 소스 | 내용 | 글자 수 | 라이선스 |
|------------|------|---------|----------|
| **Unicode Unihan** `kKorean` | CJK 한자의 한국어 독음 | ~27,000자 | Unicode License (무료) |
| 우리 `char_map` | 평측 + 운부 | 8,550자 | 자체 |
| 우리 `poems.full.json` | 시 원문 320편 | - | 자체 |

**구축 방법**: Unihan `kKorean`과 우리 char_map을 교차(intersect)하여, **시에 관련된 한자만** 추출 + 독음 + 평측 + 운부 통합 DB 생성

```
결과 예시:
{
  "가": [
    {"char": "佳", "tone": "平", "rhyme": "上平九佳", "freq": 12, "meanings": ["아름다울"]},
    {"char": "嘉", "tone": "平", "rhyme": "下平六麻", "freq": 5, "meanings": ["아름다울", "기릴"]},
    {"char": "家", "tone": "平", "rhyme": "下平六麻", "freq": 45, "meanings": ["집"]},
    {"char": "歌", "tone": "平", "rhyme": "下平五歌", "freq": 18, "meanings": ["노래"]},
    ...
  ]
}
```

**2) 시어 빈도 DB**

`poems.full.json` 320편의 모든 한자를 카운트:
- 각 글자별 출현 횟수 → 빈도 순위
- **이미 있는 데이터**로 스크립트 하나면 생성 가능
- 예상: 약 2,500~3,000종 한자 (당시삼백수에 사용된 고유 글자)

**3) 훈음(訓音) 데이터**

- Unihan `kDefinition` (영어 뜻) + `kKorean` (독음)
- 또는 한국어 훈음 별도 DB 필요 (예: "아름다울 가", "집 가")
- 국립국어원 한자 DB나 한국어 Wiktionary에서 수급 가능

### 4-3. 웹 IME 아키텍처

```
┌──────────────────────────────────────────┐
│              원고지 셀 (input)            │
│  ┌──────┐                                │
│  │ [가] │ ← 한글 입력 중 (composing)      │
│  └──┬───┘                                │
│     │ compositionend 이벤트               │
│     ▼                                    │
│  ┌──────────────────────────────────┐    │
│  │        후보 패널 (dropdown)        │    │
│  │  ┌────┬──────┬────┬────┬──────┐ │    │
│  │  │ 佳 │아름다울│ 平 │ ●  │ 12회 │ │    │
│  │  │ 歌 │노래   │ 平 │ ●  │ 18회 │ │    │
│  │  │ 家 │집     │ 平 │ ●  │ 45회 │ │    │
│  │  │ 嫁 │시집갈 │ 仄 │ ○  │ 2회  │ │    │
│  │  └────┴──────┴────┴────┴──────┘ │    │
│  │  [← 1/3 →]    [힌트 🔍]         │    │
│  └──────────────────────────────────┘    │
└──────────────────────────────────────────┘

● = 현재 자리에 필요한 평/측과 일치
○ = 불일치 (선택은 가능하나 빨강 표시될 것임)
```

### 4-4. 핵심 기술 포인트

**1) 한글 조합 감지 (Composition Events)**

```javascript
// 원고지 각 셀에 이벤트 리스너
cell.addEventListener('compositionstart', () => {
  // 한글 조합 시작 — 후보 패널 준비
});

cell.addEventListener('compositionupdate', (e) => {
  // 조합 중 — 중간 상태(ㄱ → 가 → 간) 실시간 반영 가능
  // 하지만 한 셀 = 한 글자이므로, 독음 1음절만 감지
});

cell.addEventListener('compositionend', (e) => {
  // 조합 완료 — 최종 독음으로 후보 검색
  const reading = e.data; // "가", "산", "풍" 등
  showCandidates(reading, currentPosition);
});
```

- **브라우저 호환성**: `compositionstart/update/end`는 모든 주요 브라우저 지원 (Chrome, Firefox, Safari, Edge)
- **한글 특수 케이스**: `ㄱ` → `가` → `간` 과정에서 `compositionupdate`가 여러 번 발생. 최종 `compositionend`의 `e.data`만 사용하면 안정적

**2) 후보 정렬 알고리즘**

```
1차 정렬: 현재 자리의 평측 요구와 일치하는 글자 우선
2차 정렬: 시어 빈도(poems.full.json 기반) 내림차순
3차 정렬: 획수 오름차순 (동점 시)
```

이 정렬이 **우리 서비스의 핵심 차별점**:
- 한국어 OS IME: "가" → 家(집), 價(값), 可(옳을)... (일상 빈도순)
- 우리 IME: "가" → 佳(아름다울), 歌(노래), 家(집)... (**시어 빈도순 + 평측 필터**)

**3) 성능 고려**

- 독음→한자 DB 크기: 약 8,550자 × (독음 + 평측 + 운부 + 빈도 + 훈) ≈ **500KB~1MB JSON**
- 브라우저 메모리에 한 번 로드 → 이후 검색은 **O(1) Map lookup**
- 후보 정렬: 최대 수십 개 → 정렬 비용 무시 가능
- **결론: 성능 문제 없음**

### 4-5. 구현 난이도 평가

| 구성요소 | 난이도 | 설명 |
|---------|--------|------|
| 독음→한자 DB 구축 (Unihan 파싱 + char_map 병합) | ★★☆☆☆ | 스크립트 작업, Unihan 데이터는 공개 |

### 5-2. 106운목 테이블 레이아웃

형 아이디어: "원소 주기율표처럼"

```
┌─────────────────────────────────────────────┐
│  평성 (30운부)                               │
│  ┌───┬───┬───┬───┬───┬───┬───┬───┬───┬───┐ │

│                                              │
│  측성 (76운부) — 접힌 상태, 펼침 가능          │
│  [상성 29] [거성 30] [입성 17]               │
└─────────────────────────────────────────────┘
```

**구현 고려사항**:
- 압운은 **평성 30운부**에서만 선택하는 게 정격 (측성 운 사용은 드묾)
- 따라서 평성 30운부를 **기본 표시**, 측성 76운부는 접힌 상태
- 각 운목 셀에 **훈음 항상 표시** (형 결정)
- 운목 클릭 → 아코디언 펼침 → 소속 글자 표시

### 5-3. 소속 글자 표시 문제

운부마다 글자 수가 크게 다름:
- 적은 운부: 20~30자 (예: 上平三江)
- 많은 운부: 200자 이상 (예: 上平四支)

**해결 방안**:
- 아코디언 내부에 **격자형 나열** (한 줄에 10~15자)
- **검색/필터** 추가: 독음으로 필터링 가능
- 글자에 **호버/탭 시 훈음** 표시
- **평측 태그**: 다음자의 경우 이 운부에서의 성조 표시

**구현 난이도**: ★★☆☆☆ (낮음) — 데이터가 이미 있으므로 UI 렌더링만 하면 됨

## 10. 결론 및 다음 단계

### 기술적으로 구현 가능한가?

**예.** 치명적 기술 장벽은 없다.

- 엔진: 규칙이 명확하고 데이터가 이미 있음
- IME: 웹 표준 API로 구현 가능, 핵심 데이터(Unihan)도 공개
- AI: 모델 선택지가 충분하고 비용도 합리적
- UI: 기존 프로젝트 패턴 재사용 가능

### 핵심 기술 리스크

1. **훈음 DB 소싱**: 한국어 "훈음" 데이터가 정돈된 형태로 공개된 소스가 제한적. Unihan `kDefinition`은 영어. 한국어 훈음은 별도 수급 또는 구축 필요.
2. **다음자 목록 완성도**: 평수운에서 다음자를 얼마나 정확히 식별하느냐가 엔진 신뢰도에 직결. 학술 자료 참조 필요.
3. **AI 프롬프트 품질**: 시어 추천의 품질은 프롬프트 설계에 크게 의존. 반복 테스트/튜닝 필요.

### 제안하는 구현 순서

```
Phase 1: 데이터 기반 구축
  → char_map 확장 (다음자 + 입성)
  → 시어 빈도 DB 생성
  → 독음→한자 DB 구축

Phase 2: 엔진 코어
  → 평측 판정 + 격률 검사 + 병 탐지
  → 신호등 로직

Phase 3: UI (원고지 + 운목)
  → 원고지 격자 UI
  → 운목 선택 아코디언
  → 한글-한자 IME

Phase 4: AI 연동
  → Supabase Edge Function 설정
  → 시어 추천 (힌트)
  → 번역 확인 + 퇴고 코칭

Phase 5: 통합 + 5단계 플로우
  → 1~5단계 전체 흐름 연결
  → Path A/B 분기
```
