## 1. 메인페이지 재설계

### 현재 상태
- 현 index.html이 곧바로 타임라인 (당시삼백수 열람 페이지)
- tang-poetry-300/ 에 별도 시집 페이지 존재 (미정리)

### 수정 방안

**페이지 구조 변경:**
```
index.html (새 메인 = 포털)
├── A. 히어로 섹션
├── B. 오늘의 시
├── C. 서브페이지 소개 (4개)
├── D. 한시 도우미 소개
└── E. 푸터

/tang-poetry-300/ → 기존 타임라인 이동 (또는 새 경로)
```

#### A. 히어로 섹션
- **구현 방식**: `river_scroll.jpg` (5257×1408)를 CSS `@keyframes`로 천천히 가로 스크롤
- **보트 영상**: `rawingboat_1.webm` (알파 webm)을 `<video>` 태그로 루프, `position: absolute`로 이미지 위에 배치
- **안개 효과**: CSS gradient 애니메이션 또는 canvas 기반 파티클 — 하단에서 위로 올라오는 반투명 흰색 레이어
- **히어로 카피**: `samusa.png`를 중앙에 `position: absolute`로 배치
- **기술적 주의**: webm 알파는 Safari 미지원 → `<picture>` 또는 fallback 이미지 필요

#### B. 오늘의 시
- poems.full.json에서 배경그림 있는 시를 필터링 → 매일 랜덤 3~5수 표시
- 형이 포토샵으로 만든 이미지 위에 시인명+제목만 오버레이
- 클릭 시 해당 시 모달 팝업 (기존 openPoemModal 재활용)

#### C. 서브페이지 소개
- 카드 4개: 당시삼백수(타임라인), 시인(poets), 시(poem), 역사(history)
- 각 카드에 대표 이미지 + 한 줄 설명 + 링크

#### D. 한시 도우미 소개
- 4번 작업 완성 후 추가 (MVP 콘텐츠이므로 별도 소개 섹션)

### 형이 결정/준비할 것
1. **타임라인 페이지 URL 확정**: `/tang-poetry-300/`으로 이동? 아니면 `/timeline/`?
2. **오늘의 시 이미지**: 포토샵 작업한 시+그림 이미지 파일들 준비 (몇 수 분량?)
3. **히어로 카피 (samusa.png)**: 최종본 확인
4. **서브페이지 소개 문구**: 각 페이지별 1~2줄 설명 텍스트

### 워크플로우
```
1. [형] URL 구조 확정 + 오늘의 시 이미지 준비
2. [민철] 기존 타임라인을 새 경로로 이동 (라우팅 정리)
3. [태훈] 히어로 섹션 애니메이션 구현 (스크롤+영상+안개)
4. [민철] 오늘의 시 로직 + 서브페이지 소개 섹션
5. [태훈] 히어로 반응형 + 성능 최적화
6. [형] 최종 확인 + 피드백
```

### 태훈 협업 가능 부분
- **히어로 섹션 애니메이션 전체**: 가로 스크롤 + webm 오버레이 + 안개 파티클은 프론트 애니메이션 전문 영역 → 태훈에게 적합
- **성능 최적화**: 5257px 이미지 + 영상 동시 로딩 시 LCP 최적화 필요

## 2. 서브페이지 디자인 수정 + readdy 콘텐츠

### 현재 상태
- `/pages/`, `/poem/`, `/poets/`, `/history/` — HTML 뼈대만 존재
- 실제 콘텐츠는 `https://mvudfy.readdy.co` 에 있음 (형이 만든 콘텐츠, readdy 플랫폼에 업로드됨)

### 수정 방안
1. **readdy 콘텐츠 수집**: fetch로 해당 URL에서 텍스트/이미지 추출 → 로컬 JSON/MD로 정리
2. **각 페이지 디자인**: 현재 타임라인 v3 톤앤매너(크림+고전서체+금색포인트) 유지
3. **콘텐츠 배치**: 정리된 텍스트를 각 페이지 HTML에 삽입

### 형이 결정/준비할 것
1. **readdy 콘텐츠 범위**: 어떤 페이지의 어떤 콘텐츠를 가져올지 목록
2. **각 페이지 우선순위**: 4개 페이지 중 먼저 작업할 순서
3. **디자인 방향**: 현재 톤 유지? 피그마에서 새로 디자인?

### 워크플로우
```
1. [형] readdy에서 가져올 콘텐츠 목록 + 페이지별 우선순위 정리
2. [태훈] readdy URL fetch → 콘텐츠 크롤링 + JSON/MD 정리
3. [민철] 각 페이지 레이아웃 + 콘텐츠 삽입
4. [형] 콘텐츠 확인 + 수정 요청
```

### 태훈 협업 가능 부분
- **readdy 콘텐츠 크롤링/정리**: 웹에서 텍스트+이미지 추출 → 구조화된 파일로 정리
- **서브페이지 디자인 시안**: 피그마 또는 HTML 프로토타입

## 3. TTS (Qwen3-TTS)

### 현재 상태
- 조사 완료: Qwen3-TTS (0.6B/1.7B), Apache 2.0, 중국어+한국어 지원
- 데모 테스트 아직 안 함

### 수정 방안
1. **데모 테스트**: HuggingFace Space에서 시 낭송 품질 확인
2. **품질 확인 후 일괄 생성**: 클라우드 GPU에서 313수 생성 → `public/audio/` 저장
3. **사이트 연동**: 기존 TTS 버튼 (`tts-btn`) 에 오디오 파일 연결

### 형이 결정/준비할 것
1. **음성 스타일**: 학자풍? 남성/여성? 속도? → 데모 테스트 후 결정
2. **예산 승인**: 클라우드 GPU 비용 $5~20 (스팟 인스턴스)

### 워크플로우
```
1. [지훈] HuggingFace 데모에서 샘플 5~10수 생성 → 형에게 공유
2. [형] 음성 품질/스타일 확인 + 피드백
3. [지훈] 클라우드 GPU 환경 세팅 (A10G/T4 스팟)
4. [지훈] generate_tts.py 스크립트로 313수 일괄 생성
5. [민철] 생성된 오디오 파일을 사이트에 연동
```

### 지훈 협업 — 이 작업은 지훈이 전담 적합
- **GPU 서버 세팅 + 모델 로딩**: Python + PyTorch 환경
- **일괄 생성 스크립트**: poems.full.json에서 poemZh 추출 → 오디오 변환
- **품질 관리**: 생성 후 무음/잡음/길이 이상 자동 체크
- 이미 `scripts/generate_tts.py` 파일이 존재 (뼈대 있음)

## 4. 한시 작성 도우미

### 현재 상태
- 기획 완료: GPT와 대화 → 지훈이 3개 설계 문서 작성 (07/08/09번)
- 구현: 아직 시작 안 함
- 핵심 구조: 학습(규칙설명) → 예시(대표작 하이라이트) → 실습(7단계 도우미)
- 엔진/AI 분리 원칙 확립

### 수정 방안 (단계적 접근)

**Phase 1 — 엔진 코어 (지훈)**
- 운서 DB 구축 (평수운: 한자→운부, 운부→한자목록)
- 평측 DB 구축 (한자→평/측)
- 율격 템플릿 (오언/칠언 × 절구/율시, 평기/측기)
- 검사기: 형식검사, 운검사, 평측검사, 병탐지

**Phase 2 — 기본 UI (민철)**
- 7단계 Stepper UI
- 형식 선택 → 운목 선택 → 입력 → 일괄 검사 (모드 A만 먼저)
- 엔진 출력을 "학습형 리포트"로 표시

**Phase 3 — 실시간 모드 (민철 + 지훈)**
- 모드 B: 글자별 입력 + 신호등
- 운서 검색 패널

**Phase 4 — AI 연동 (지훈)**
- 운자 추천 (주제/정서 기반)
- 구제 후보 추천 (요체 처리)
- 구상 vs 완성 비교 리포트

### 형이 결정/준비할 것
1. **평측/운부 기준 확정**: 교수님과 협의 필요 (평수운? 광운?)
2. **대표 작품 선정**: 각 시체별 2~4편 (학습 영역용)
3. **요구(拗救) 지원 범위**: 초급만? 중/고급까지?
4. **AI 과금 정책**: 무료 N회 → 구독? 아니면 전체 무료?
5. **지훈이 설계문서(07/08/09) 타당성 판단**: 이대로 갈지 수정할지

### 워크플로우
```
Phase 1 (엔진 — 2~3주)
1. [형] 교수님과 평측/운부 기준 협의
2. [지훈] 운서 DB + 평측 DB 구축 (scripts/build_pingshui_db.js 이미 존재)
3. [지훈] 율격 템플릿 + 검사기 구현
4. [지훈] 단위 테스트: 당시삼백수 10수 대상 검사 결과 검증

Phase 2 (기본 UI — 1~2주)
5. [민철] 7단계 Stepper + 입력 UI 구현
6. [민철] 엔진 연동 + 검사 리포트 화면
7. [형] 테스트 + 피드백

Phase 3 (실시간 모드 — 1~2주)
8. [민철] 글자별 입력 UI + 신호등
9. [지훈] 실시간 검증 API (글자 입력마다 엔진 호출)

Phase 4 (AI — 1~2주)
10. [지훈] AI 프롬프트 설계 + API 연동
11. [민철] AI 추천 UI (운자 후보, 구제 후보)
12. [형] 전체 테스트 + 최종 확인
```

### 지훈 협업 — 이 프로젝트의 핵심 담당
- **엔진 전체**: 운서DB, 평측DB, 검사기, 요체처리 — 지훈이 설계했으므로 본인이 구현하는 게 가장 효율적
- **AI 연동**: OpenAI/Claude API 호출 + 프롬프트 설계
- 민철은 **UI/프론트만 담당**하고 엔진은 지훈에게 맡기는 구조가 깔끔

## 5. 모바일 반응형

### 현재 상태
- 미착수
- 현재 타임라인 v3은 데스크탑 전용 (3컬럼 그리드)

### 수정 방안
- `@media` 브레이크포인트 설정: 768px (태블릿), 480px (모바일)
- **타임라인**: 3컬럼 → 1컬럼 (시인카드 → 타임라인선 → 설명 순서로 세로 배치)
- **시인카드**: 가로 스크롤 또는 2열 축소
- **시 모달**: 2컬럼(한자|한글) → 1컬럼 (한자 위, 한글 아래)
- **관리툴**: 관리용이므로 모바일 대응 후순위

### 형이 결정/준비할 것
1. **우선순위**: 메인(타임라인) 먼저? 모달 먼저?
2. **모바일 네비게이션**: 햄버거 메뉴? 탭 바?

### 워크플로우
```
1. [민철] 타임라인 v3 반응형 (3컬럼→1컬럼)
2. [민철] 시인 모달 / 시 모달 반응형
3. [민철] 헤더/푸터/네비게이션 반응형
4. [형] 실기기 테스트 + 피드백
```

### 협업
- 민철 단독 작업 가능 (CSS 미디어쿼리 중심)
- 태훈이 히어로 섹션 작업 시 동시에 반응형 포함하면 효율적
