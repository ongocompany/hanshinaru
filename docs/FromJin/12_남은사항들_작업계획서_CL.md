## 1. 메인페이지 재설계

### 현재 상태
- `index.html` + `app.js` + `styles.css` 가 곧바로 타임라인 페이지
- `tang-poetry-300/index.html`에 별도 시집 페이지 존재 (미정리)

### 수정 방안

**페이지 구조 변경 (확정):**
```
index.html (새 메인 = 포털)          ← 완전히 새로 작성
├── A. 히어로 섹션
├── B. 오늘의 시 (플레이스홀더)
├── C. 서브페이지 소개 (4개 카드)
├── D. 한시 도우미 소개 (나중에)
└── E. 푸터

/timeline/index.html                  ← 기존 타임라인 이동 (app.js + styles.css 포함)
```

#### A. 히어로 섹션

**에셋 파일:**
| 파일 | 경로 | 설명 |
|------|------|------|
| 배경 이미지 | `docs/reference/메인페이지히어로섹션/river_scroll.jpg` | 5257×1408px, 가로 스크롤용 |
| 보트 영상 | `docs/reference/메인페이지히어로섹션/rawingboat_1.webm` | 알파값 포함 WebM |
| 히어로 카피 | `docs/reference/메인페이지히어로섹션/samusa.png` | 중앙 배치 |

**구현 방식:**
- `river_scroll.jpg` → CSS `@keyframes`로 천천히 가로 스크롤
- `rawingboat_1.webm` → `<video loop muted autoplay>` + `position: absolute`로 배경 위에
- 안개 효과 → CSS gradient 또는 canvas 파티클, 하단→상단 반투명 흰색
- `samusa.png` → 중앙 `position: absolute`
- ⚠️ Safari: webm 알파 미지원 → fallback 이미지 필요

#### B. 오늘의 시
- **데이터 소스**: `public/index/poems.full.json`에서 랜덤 3~5수
- **이미지**: 우선 플레이스홀더 카드로 구현 → 형이 포토샵 이미지 준비되면 교체
- **클릭 동작**: 해당 시 모달 팝업 (기존 `app.js`의 `openPoemModal()` 재활용)

#### C. 서브페이지 소개
- 카드 4개: 당시삼백수(`/timeline/`), 시인(`/poets/`), 시(`/poem/`), 역사(`/history/`)
- 각 카드에 대표 이미지 + 한 줄 설명 + 링크

#### D. 한시 도우미 소개
- 4번 작업 완성 후 추가

### 워크플로우
```
1. [민철] 기존 타임라인을 /timeline/로 이동 (파일 복사 + 경로 정리)

   - river_scroll.jpg 가로 스크롤 + rawingboat_1.webm 루프 + 안개 + samusa.png
4. [민철] 오늘의 시 (플레이스홀더) + 서브페이지 소개 섹션
```

## 2. 서브페이지 디자인 수정 + readdy 콘텐츠

### 현재 상태

**기존 서브페이지 파일:**
| 페이지 | 경로 | 상태 |
|--------|------|------|
| pages | `pages/readdy_like.css`, `pages/readdy_like.js` | CSS/JS만 존재 |
| poem | `poem/index.html` | HTML 뼈대만 |
| poets | `poets/index.html` | HTML 뼈대만 |
| history | `history/index.html` | HTML 뼈대만 |

- 실제 콘텐츠는 `https://mvudfy.readdy.co` 에 있음 (형이 만든 콘텐츠)

### 수정 방안
1. readdy에서 **본문 텍스트만** 추출 → `docs/content/` 또는 `public/content/`에 JSON/MD로 저장
2. 각 페이지 디자인: 현재 톤앤매너(크림+고전서체+금색) 유지
3. 정리된 콘텐츠를 각 페이지 HTML에 삽입

### 워크플로우
```
1. [태훈] readdy URL(https://mvudfy.readdy.co) fetch → 본문만 크롤링
   - 저장 위치: docs/content/ 또는 public/content/ (새로 생성)
   - 형식: 페이지별 JSON 또는 Markdown
2. [민철] 각 페이지 레이아웃 구현
   - 스타일 참고: styles.css (CSS 변수 시스템 사용)
   - 폰트: ui_settings.json의 fontSets 참조
3. [민철] 콘텐츠 삽입
4. [형] 콘텐츠 확인 + 수정 요청
```

### 태훈 작업 시 참고
- readdy URL: `https://mvudfy.readdy.co`
- 범위: **본문 콘텐츠만** (사이드바, 메타데이터, 헤더/푸터 제외)
- 크롤링 결과 저장: `docs/content/` 폴더 새로 만들어서 페이지별 정리

## 3. TTS (Qwen3-TTS)

### 현재 상태
- 조사 완료: Qwen3-TTS (0.6B/1.7B), Apache 2.0, 중국어+한국어 지원
- 데모 테스트 아직 안 함
- 기존 오디오: `public/audio/` (642개 파일 이미 존재)

### 관련 파일
| 파일 | 경로 | 설명 |
|------|------|------|
| TTS 생성 스크립트 | `scripts/generate_tts.py` | Python, 뼈대 존재 |
| 시 데이터 | `public/index/poems.full.json` | `poemZh` 필드에서 원문 추출 |
| 오디오 저장소 | `public/audio/` | 생성된 mp3/wav 저장 위치 |
| TTS 테스트 페이지 | `test_tts.html` | 브라우저에서 테스트용 |
| 사이트 연동 | `app.js` | `tts-btn` 클래스의 버튼 이벤트 |

### 워크플로우
```
1. [지훈] HuggingFace 데모에서 샘플 5~10수 생성 → 형에게 공유
   - 참고: https://discuss.pytorch.kr/t/qwen3-tts-500-12hz-omni-audio/8817
2. [형] 음성 품질/스타일 확인 + 피드백
3. [지훈] 클라우드 GPU 환경 세팅 (A10G/T4 스팟)
4. [지훈] scripts/generate_tts.py 수정 → 313수 일괄 생성
   - 입력: public/index/poems.full.json (poemZh 필드)
   - 출력: public/audio/ 폴더에 시 ID별 파일명
5. [민철] app.js에서 생성된 오디오 파일 연동 (tts-btn 이벤트)
```

### 지훈 작업 시 참고
- `scripts/generate_tts.py` — 이미 뼈대 코드 있음, 여기서 이어서 작업
- `public/index/poems.full.json` — 각 시 객체의 `poemZh` 필드가 중국어 원문
- `public/audio/` — 이미 642개 파일 존재 (기존 것과 네이밍 규칙 맞추기)
- `test_tts.html` — 브라우저에서 생성 결과 확인 가능

### 수정 방안 (단계적 접근)

**Phase 1 — 엔진 코어 (지훈)**
- 운서 DB: `public/index/pingshui_yun.json` 이미 있음 → 구조 확인 후 보강
- 평측 DB: `scripts/generate_pinyin_pingze.py`로 생성 → JSON 저장
- 율격 템플릿: 오언/칠언 × 절구/율시, 평기/측기
- 검사기: 형식검사, 운검사, 평측검사, 병탐지
- **평측/운부 기준**: 평수운 기준으로 진행 (현실적 수준, 광운까지 안 감)
- **요구(拗救)**: 기본적인 것만 (초급 수준)

**Phase 2 — 기본 UI (민철)**
- 7단계 Stepper UI
- 형식 선택 → 운목 선택 → 입력 → 일괄 검사 (모드 A)
- 엔진 출력을 "학습형 리포트"로 표시

**Phase 3 — 실시간 모드 (민철 + 지훈)**
- 모드 B: 글자별 입력 + 신호등
- 운서 검색 패널

**Phase 4 — AI 연동 (지훈)**
- 운자 추천, 구제 후보 추천, 비교 리포트

### 워크플로우

### 서비스 선정 근거

**Supabase 선택 이유 (vs Firebase, VPS):**
- PostgreSQL 기반 → SQL 유연성 + 풀텍스트 검색 내장
- 무료 티어: DB 500MB, 스토리지 1GB, 인증 무제한, API 무제한
- 유료: $25/월 (Firebase보다 예측 가능한 과금)
- 오픈소스 → 나중에 자체호스팅 전환 가능 (vendor lock-in 없음)
- VPS 대비 서버 관리 부담 없음 (보안/백업/업데이트 불필요)

**서버리스 GPU 선택 이유 (vs 상시 GPU 서버):**
- 상시 GPU 서버: 월 $100~300 (사용자 적으면 낭비)
- 서버리스: 사용할 때만 과금 (시 1수 생성 ~$0.01~0.03)
- RunPod: T4 ~$0.20/hr, A10G ~$0.40/hr, 초당 과금
- Qwen3-TTS CustomVoice: 3초 음성 샘플로 목소리 복제 가능

### TTS 두 가지 시나리오

**시나리오 1 — 기본 TTS (미리 생성):**


**시나리오 2 — "내 목소리로 읽어주는 한시" (실시간):**
```

담당: 지훈(GPU 배포) + 민철(프론트 연동)
```

### AI 기사 재작성 규칙

- **분량**: 300~500자 (한국어)
- **톤**: 사이트 톤앤매너에 맞게 — 교양 있고 차분, 한문/시 문화에 친화적
- **형식**: 기사 제목 + 본문 + 원문 출처 링크
- **저작권**: 원문 요약이 아니라 사실 기반 **재작성** → 저작권 문제 회피
- **AI 사용**: Claude API 또는 GPT API (기사당 ~$0.01 이하)
- **필터링**: AI가 "한시/한자/전통문화에 관련 있는가?" 판별 → 무관한 기사 제거

### 프론트엔드 UI (메인페이지)

**방식: 아코디언 UI** (데이터 볼륨 월 8~12건이므로 별도 게시판 불필요)

- 최신 **5~8개** 기사를 메인페이지 "한시관련 소식" 섹션에 표시
- 기사 제목 클릭 → 아코디언으로 본문 펼침
- 카테고리 태그 표시 (한시, 한자, 시문학, 전통행사, 서예 등)
- **오래된 기사**: 2개월 경과 시 화면에서 제거, JSON 아카이브에만 보존
- 향후 데이터 볼륨 증가 시 별도 `/news/` 페이지로 확장 가능

### GitHub Actions 워크플로우

```yaml
# .github/workflows/crawl-news.yml
name: Crawl Poetry News
on:
  schedule:
    - cron: '0 9 * * 1'   # 매주 월요일 오전 9시 (UTC)
  workflow_dispatch:        # 수동 실행 가능

jobs:
  crawl:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
      - run: npm install
      - run: node scripts/crawl_news.js
        env:
          NAVER_CLIENT_ID: ${{ secrets.NAVER_CLIENT_ID }}
          NAVER_CLIENT_SECRET: ${{ secrets.NAVER_CLIENT_SECRET }}
          AI_API_KEY: ${{ secrets.AI_API_KEY }}
      - run: |
          git config user.name "GitHub Actions"
          git config user.email "actions@github.com"
          git add public/index/news_articles.json
          git commit -m "[Auto] 한시 소식 업데이트" || true
          git push
```
