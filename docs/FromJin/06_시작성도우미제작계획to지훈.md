## 1) 페이지 IA(정보 구조) 제안

### A. 상단: 근체시 안내(읽는 영역)

- **근체시 개요**(형식/압운/평측/대우/금기/예외) — 아까 정리한 “설명문 버전” 그대로 본문으로 들어감
- 그 아래에 “대표 작품으로 규칙 보기” 섹션을 붙임

### B. 중단: 시체별 대표 작품 + 규칙 하이라이트(예시 학습 영역)

- 탭/드롭다운: `오언절구 / 칠언절구 / 오언율시 / 칠언율시 / 배율`
- 각 탭에서 대표작 2~4편 정도를 카드로 보여주고, 클릭하면:
    - **압운 글자(2·4·6·8구)** 자동 하이라이트
    - *평측 패턴(가능하면)**을 구별 표시(초급은 “정격/요구 여부”만)
    - *대우 구간(3–4 / 5–6)**은 두 줄을 묶어서 “대구 대응” 표시(품사/구조 설명 툴팁)
- 이 영역이 중요한 이유: 사용자가 “규칙 텍스트 → 실제 시의 작동”을 즉시 연결함

### C. 하단: “직접 써보기” 입력 도우미(실습 영역)

- 여기서 **모드 선택**이 핵심
    1. **자유 작성 + 일괄 검사 모드**(후검사형)
    2. **가이드 작성 + 실시간 신호등 모드**(동행형)

## 2) 입력 도우미 7단계 UI를 실제 화면으로 번역

말씀하신 7단계를 그대로 유지하되, “사용자가 지치지 않게” 각 단계의 출력물을 다음 단계에 넘기도록 설계하면 좋습니다.

### Step 1. 형식 정하기

- 선택: 오언/칠언 × 절구/율시/배율
- 옵션(고급): 평기/측기, 수구입운(1구 입운) 여부
- 여기서 **템플릿(몇 구, 어디 압운)**이 자동 세팅됨

### Step 2. 주제·소재 적기

- 입력: 주제(1줄), 소재/이미지(키워드 5~10개), 정서 톤(선택)
- 여기서 AI를 쓰고 싶다면 “정서/이미지의 결”을 정리해주는 정도는 유용함
    - 단, **시어를 만들어주지는 않음**(사용자 창작 영역 침범 방지)

### Step 3. 한글로 각 구에 들어갈 내용 정리해보기(개요 설계)

- 8구(또는 4구) 칸을 한글로 채우는 “스토리보드”
- 율시는 련별 기능 안내를 오른쪽에 고정:
    - 1련(제시), 2련(대우1), 3련(대우2), 4련(결구)
- 여기까지 하면 사용자는 이미 “무엇을 말할지”를 확보한 상태

### Step 4. 운목 정하기(운부/운자 선택)

- UX 핵심: **운자를 고르기 전에 운부 후보를 제시**하는 것
- AI 사용 포인트(여기서 제일 자연스러움):
    - 사용자가 주제/소재를 쓰면 AI가 “어울리는 운의 분위기”를 설명하고,
    - 후보 운부/후보 운자(예: 6~12개)를 “추천”해줌
- 다만 최종 결정은 사용자 + 운서 DB 기반으로 확정

### Step 5. 압운 자리 채우기(2·4·6·8구)

- 사용자가 운자를 고르면, 시스템은:
    - 해당 운부에서 “운자 후보 리스트”를 보여주고
    - 압운 자리(2/4/6/8구 말미)에 넣게 함
- **AI 개입 없이도 충분히 만족**이 나오는 구간(운서 검색이 곧 UX)

### Step 6. 나머지 부분 채워 넣기(핵심 창작)

여기서 두 갈래 모드가 갈립니다.

### 모드 A: “내가 다 쓰고, 한 번에 검사”

- 입력창을 넓게 주고, 다 쓰면 `학인(검사)` 버튼
- 결과는 “정답지처럼” 나오면 안 되고, **어디가 왜 문제인지 설명형**으로 나와야 학습이 됨

### 모드 B: “한 글자씩 가이드 받으며 쓰기”

- 각 구를 **토큰 단위(글자 단위)**로 입력하는 UI(중국어 입력 보조랑 비슷한 느낌)
- 하단 추천 패널:
    - “운서 기반 후보”(AI 없음)
    - “의미 기반 후보”(AI 옵션)
- 실시간 신호등:
    - **빨강**: 구조적으로 불가능(예: 압운 자리 불일치, 글자수 초과, 운부 이탈)
    - **노랑**: 가능하지만 병(고평/삼평미 등) 위험, 대우 균형 흔들림 경고
    - **초록**: 형식상 문제 없음
- 이 모드의 장점은 “시를 쓰는 손맛”이 생기고, 초보가 좌절하지 않습니다.

### Step 7. 감상하기

- 완성본을 “작품 카드”로 예쁘게 보여주고,
- 하단에 “형식 리포트”를 접었다 펼 수 있게 둠
- 가능하면 “초급 요약”과 “상세 리포트”를 나눠서 제공

## 3) ‘AI는 어디에 쓰는 게 딱 좋나’ (과잉개입 방지 설계)

형이 제안한 철학(운서 도움은 하되 창작 침범 최소화)을 유지하면서, AI 효용이 큰 지점만 찍으면 이렇습니다.

### AI가 효과 큰 지점

1. **Step 2–3**: 주제/소재를 “시적으로 압축된 이미지 묶음”으로 정리(요약/재배열)
2. **Step 4**: 주제/정서에 맞는 **운부/운자 후보 추천 + 분위기 설명**
3. **모드 B에서 옵션**: 사용자가 이미 의미를 정한 상태에서, 그 의미에 맞는 “후보 시어(단어)”를 제시
    - 단, 자동 완성처럼 보이면 창작이 아니라 “작성 보조”가 되니, 항상 “후보”로만

### AI가 굳이 필요 없는 지점(오히려 빼는 게 낫다)

- 압운/평측/대우의 **형식 판정**
- 운서에서 **동일 운부 글자 검색**
- 병(고평 등) 탐지
    
    이건 전부 룰 엔진이 더 신뢰성 있습니다.

## 4) 구현 관점에서의 핵심 엔진 분리(안정성)

이 서비스는 사실 “UI”보다 아래 3개가 성패를 가릅니다.

1. **운서 DB / 운부 매핑**
    - 평수운 기반으로: `한자 → 운부`, `운부 → 한자 리스트`
2. **평측 DB**
    - `한자 → 평/측` (상거입 포함)
3. **율격 템플릿 + 검사기**
    - 형식(자수/구수), 압운 위치, 평측 패턴, 대우 구간, 병 규칙(고평 등)

그리고 **AI는 위 엔진 결과를 “설명하는 말”을 잘해주는 역할**로 붙이면 됩니다.

즉, “판정은 코드”, “코치 멘트는 AI” 구조가 가장 안전합니다.

- “일괄 검사”는 **창작 자유 + 학습 피드백**에 강하고,
- “실시간 신호등”은 **초보자 진입장벽을 낮추고 규칙을 몸으로 익히게** 합니다.

> 목적: 근체시(절구/율시/배율) 규칙을 학습하고, 입력 도우미를 통해 사용자가 규칙 기반으로 창작을 경험하도록 한다.

> 핵심 원칙: **형식 판정은 엔진(규칙 기반, deterministic)**, **언어적 보조(추천/진단/설명)는 AI(선택적, 후보 제시)**로 분리한다.

## 0. 전체 결론 요약 (우리 대화에서 합의된 방향)

- 근체시 창작 UX는 **두 모드**가 함께 있을 때 학습 효과가 크다.
    1. **자유 작성 + 일괄 검사(후검사형)**: 창작 자유를 보장하고, 결과 리포트로 학습.
    2. **가이드 작성 + 실시간 신호등(동행형)**: 초보 진입장벽을 낮추고, 규칙을 몸으로 익힘.
- 압운/평측/형식 오류는 **엔진**이 100% 신뢰 가능한 판정/좌표를 제공한다.
- **대우(對偶)**, **요체(拗體/拗救)**는 엔진만으로 한계가 있으므로:
    - **엔진이 구조/좌표/가능 여부를 확정**하고,
    - **AI가 후보/수정안/진단 설명**을 제공하는 하이브리드가 최적.
- AI는 “정답 판정자”가 아니라 “코치” 역할로 제한한다. 자동 적용은 지양하고, 항상 **후보 제시 + 사용자 선택**을 기본으로 한다.

## 1. 페이지 IA(정보 구조) / UI 큰 구조

### 1) 상단: 근체시 안내(읽는 영역)

- 근체시 개요(형식/압운/평측/대우/금기/예외(요구)) 설명문 제공
- 규칙 텍스트 → 실제 예시로 연결되도록 아래 “대표 작품” 섹션으로 자연스럽게 유도

### 2) 중단: 시체별 대표 작품 + 규칙 하이라이트(예시 학습 영역)

- 탭/드롭다운: `오언절구 / 칠언절구 / 오언율시 / 칠언율시 / 배율`
- 각 탭에 대표작 2~4편 카드
- 작품 클릭 시 오버레이/하이라이트:
    - 압운 글자(2·4·6·8…구) 강조
    - 대우 구간(율시: 3–4, 5–6구) 묶어서 표시
    - 평측(가능하면) 패턴 힌트 표시(초급: 정격/요구 여부 정도)
- 목표: 사용자가 규칙을 “설명→작동 사례”로 바로 체감

### 3) 하단: “직접 써보기” 입력 도우미(실습 영역)

- **모드 선택 토글** 제공:
    - (A) 자유 작성 + 일괄 검사
    - (B) 가이드 작성 + 실시간 신호등

## 2. 입력 도우미 7단계 플로우(고정)

> 사용자가 지치지 않도록 각 단계 산출물을 다음 단계로 넘기며 진행한다.
> 

### Step 1. 형식 정하기

- 선택: 오언/칠언 × 절구/율시/배율
- 옵션(고급): 평기/측기, 수구입운(1구 입운) 여부
- 결과: 템플릿 자동 세팅(구수/자수/압운 위치/대우 구간)

### Step 2. 주제·소재 적기

- 입력: 주제(1줄), 소재/이미지 키워드(5~10개), 정서 톤(선택)
- AI(선택): 소재 재배열/압축/정서 톤 정리(창작 침범 금지)

### Step 3. 한글로 각 구의 내용 설계(스토리보드)

- 4구/8구 칸에 한글로 “말할 내용”을 미리 적는다.
- 율시는 련별 기능 힌트 고정:
    - 1련(제시), 2련(대우1), 3련(대우2), 4련(결구)

### Step 4. 운목(운부/운자) 정하기

- 운자 고르기 전에 운부 후보 제시가 핵심
- AI(효율 높음):
    - 주제/정서 기반으로 “어울리는 운의 분위기” 설명
    - 후보 운부/운자(예: 6~12개) 추천
- 최종 확정은 운서 DB/엔진 기반

### Step 5. 압운 자리 채우기(2·4·6·8…구)

- 운부 확정 후, 운부 내 운자 목록/검색 제공
- AI 없이도 충분(운서 검색 UX가 핵심)

### Step 6. 나머지 채우기(창작 핵심) — 모드 분기

### (A) 자유 작성 + 일괄 검사

- 넓은 입력창 → `학인(검사)` 버튼
- 결과는 “오류 좌표+근거” 중심의 학습형 리포트로 제시

### (B) 가이드 작성 + 실시간 신호등

- 한 글자(또는 슬롯) 단위 입력 UI
- 하단 추천 패널:
    - 운서 기반 후보(엔진) + (옵션) 의미 기반 후보(AI)
- 신호등:
    - 빨강: 구조적으로 불가(글자수 초과, 운부 이탈, 압운 자리 위반 등)
    - 노랑: 가능하나 병(고평/삼평미 등) 위험, 대우 균형 경고
    - 초록: 형식상 문제 없음

### Step 7. 감상하기

- 완성본 작품 카드로 출력
- 형식 리포트는 접기/펼치기
- “초급 요약”과 “상세 리포트” 분리 권장

## 3. 엔진/AI 역할 분리 (신뢰성 설계)

### 3.1 엔진이 책임지는 것 (deterministic, 확정 판정)

- 형식: 구수/자수/압운 위치
- 운서: `한자 → 운부`, `운부 → 한자 목록` (평수운 기반 권장)
- 평측: `한자 → 평/측` 매핑
- 율격 템플릿: 평기/측기, 입운/불입운 등 정격 패턴 관리
- 검사기:
    - 압운 일관성(환운/중운/락운/출운)
    - 평측 위반 좌표
    - 병 탐지(고평, 삼평미/삼측미 등) — 최소 세트부터 단계적 확장
    - 대우 구간 존재 여부(형식상) 및 구조적 힌트(아래 5장)

> 원칙: 엔진 출력은 “OK/NG + 위치 + 근거”가 명확해야 한다.
> 

### 3.2 AI가 담당하는 것 (선택적, 후보/진단/설명)

- Step2–3: 주제/소재 정리(창작 개입 최소화)
- Step4: 운부/운자 후보 추천(최종 확정은 엔진)
- Step6(모드B 옵션): 의미를 유지하는 “후보 시어” 추천(자동 완성 금지)
- 학인 리포트에서:
    - (요구 가능 시) 구제 자리의 후보 제안
    - (대우 문제 시) 진단 + 대안 구절 후보 생성

> 원칙: AI는 “판정자”가 아니라 “코치”. 자동 적용 X, 후보 제공 O.
>

## 4. 요체(拗體/拗救) 처리 설계 (빨간불 → 요체 전환)

### 문제 정의

- 사용자가 입력 중 빨간불(정격 위반)이 떴을 때,
    - “이 부분을 요(拗)로 인정하고, 요구(拗救)로 수습하고 싶다”는 요구가 발생한다.

### 엔진이 해야 할 일(필수)

1. 위반 지점 정확히 표시(어느 구/몇 번째 글자/기대 평측)
2. 해당 위반을 “요로 인정 가능한 유형인지” 룰로 판정
3. 요를 살릴 경우, “구제(救)가 필요한 자리 후보” 계산 및 하이라이트
4. 구제까지 반영했을 때 전체가 성립하는지 최종 확인

### AI가 도울 수 있는 일(선택)

- 엔진이 “구제 필요 슬롯”을 내주면,
    - 사용자가 유지하고 싶은 의미/이미지 조건을 받아
    - 그 슬롯에서 **평측/자수/운서 제약을 만족**하는 후보 한자/어구를 추천한다.

### UI 제안

- 빨간불 표시 옆에 버튼: `요체로 처리`
- 클릭 시:
    - 요 지점 + 구제 지점 자동 하이라이트
    - 해당 슬롯에만 `구제 후보 추천(AI)` 버튼 활성화

## 5. 대우(對偶) 판단 기준 스펙(엔진/AI 협업용)

> 대우는 “완벽 판정(Yes/No)”보다 **다차원 점검표 + 진단**이 현실적이다.
> 
> 
> 엔진은 가능한 범위를 체크하고 좌표를 제공, AI는 개선 후보와 설명을 제공한다.
> 

### 5.1 대우 평가 단위

- 기본 단위: **율시의 함련(3–4구) / 경련(5–6구)**
- 절구는 대우가 필수는 아니므로 “선택 검사”로 둔다.

### 5.2 엔진이 제공할 수 있는 ‘최소 검수 항목’(룰 기반)

아래 항목은 “대우의 품질”이 아니라 “대우가 성립하기 위한 형식적 조건/경고”다.

1. **대우 구간 존재 여부**
    - 율시: 3–4, 5–6구가 채워져 있는지
2. **자수/슬롯 구조 일치**
    - 오언: 5자, 칠언: 7자 고정
    - (선택) 내부 분절 슬롯(예: 2+3, 4+3 등) 일치 여부
3. **중복 글자 과다 반복 경고**
    - 동일 글자의 과도한 반복(대우 양쪽/연속) 탐지
4. **기초 품사 힌트(가능하면 사전 기반)**
    - 명사/동사/형용/허사(전치·조사 역할) 정도의 거친 분류
    - 완전 자동 판정보다는 “힌트”로 제시
5. **평측 대응 힌트(선택)**
    - 대우 구간에서 양쪽 구의 대응 위치에 평측이 지나치게 어긋나는지 “경고” 수준으로 제공

> 엔진 출력 형태: 체크(OK/경고) + 문제 좌표(몇 번째 글자/슬롯) + 간단 근거
> 

### 5.3 AI가 제공할 ‘대우 진단/개선’ 항목(설명+후보 생성)

AI는 아래 기준을 텍스트 진단으로 제공하고, 수정 후보를 생성한다.

1. **품사 대응**
    - 대응 위치에서 명사↔명사, 동사↔동사 등 균형 여부
2. **구조 대응**
    - 수식어+명사 ↔ 수식어+명사
    - 동사+목적어 ↔ 동사+목적어
3. **의미 범주 대응**
    - 예: 천문↔지리, 시간↔공간, 색채↔색채, 동물↔동물 등
4. **대립/교차의 긴장감**
    - 단순 동의어 나열이 아니라 대비나 교차가 생기는지
5. **허실(虛實) 균형**
    - 실사(명사·동사)와 허사의 비율/위치 균형
6. **자연스러움(문맥 적합성)**
    - 문맥 흐름을 깨지 않으면서 대구가 성립하는지

### AI 후보 생성 시 하드 제약(반드시)

- 글자수 고정(오언/칠언)
- (필요 시) 해당 자리 평측 패턴 준수
- (필요 시) 운부/압운 자리 제약 준수
- 사용자가 지정한 유지 키워드/금지 키워드 반영
- 결과는 후보 리스트로만 제공(자동 적용 금지)

### 5.4 리포트 UI에서의 표시(권장)

- 엔진: “대우 형식 체크” 결과(OK/경고/좌표)
- AI: “대우 진단 요약 + 후보 3~5개”
- CTA:
    - `4구만 대우로 고치기`
    - `대우 후보 더 보기`
    - `의미 유지 / 대비 강화 / 고전 어조` 옵션

## 1) 8장 항목(추가 조건) – 사용자/팀 결정 사항 정리

### 8-1. 대표 작품 선정 및 저작권

- 대표 작품은 **기 구축한 ‘당시삼백수 번역+원문 DB’**에서 선정한다.
- 고전 시 원문은 저작권 이슈 없음.
- 번역문은 저작권이 있으나 **공공누리 데이터**이므로 **필수 저작권 표시/출처 표기**를 따르면 문제 없음.

### 8-2. 평측/운부 기준(정책)

- 평측/운부의 “엄밀 기준(평수운/입성 처리 등)”은 서비스 정책에 직결됨.
- 이 부분은 **교수님과 협의 후 확정**(팀 내부에서 임의 확정 X).
- 확정 전까지는 엔진 설계를 “교체 가능한 정책 모듈”로 분리하는 것이 안전함:
    - (예) `tone_policy`, `rhyme_policy` 인터페이스로 추상화

### 8-3. 요구(拗救) 패턴 고급 단계 확장 의지

- 최종 목표: 초급→중급→고급으로 요구 패턴 지원 범위를 확장.
- 리스크: 요구 패턴을 “전부” 구현하려 하면 범위가 무한히 커짐.
- 해결 방향: **지원할 요구 유형을 리스트업해 단계별로 확장**하는 로드맵 채택.

### 8-4. 대우 엔진 품사 힌트(비용/정책)

- 실시간 모델 기반 품사/대우 판정은 비용(토큰) 부담이 큼.
- **사전 기반 품사 힌트(거친 수준)만으로도 초기에는 충분**할 가능성이 높음.
- AI가 개입한다면 “글자 입력 단계”가 아니라 **최종 완성(학인/감상) 단계에서 종합 평가**로 제한하는 방향이 합리적임.

## 2) 추가 요구: “구상(한글 스토리보드) vs 완성 시” 구현도 검증 + 문장 자연스러움

### 문제 정의

- 사용자는 Step 3에서 한글로 구상을 세웠으나,
    - 평측/대우 맞추느라 **도치·생략·치환**이 발생한다.
- 이때 사용자가 궁금해하는 것:
    1. **내 구상이 실제 작품에 잘 반영되었는가?**
    2. **내 한문이 실제로 말이 되는가(자연스러운가)?**

### 결론

- 이 검증은 규칙 엔진으로 불가(의미/자연스러움은 언어 판단 영역).
- **AI로 충분히 가능**하며, 비용/신뢰성 문제를 고려해 **최종 단계 이벤트에서만 호출**하는 것이 권장됨.

### 권장 UX 출력(검증 레이어)

- (A) **검증용 직역**: 각 구를 구조대로 직역, 생략된 성분은 대괄호로 보충
- (B) **자연스러운 의역**: 읽기 쉬운 한국어 풀이(문학성 필수 아님)
- (C) **핵심 어구 주석**: 도치/생략/중요 결구 포인트 해설

### “구상 vs 완성” 리포트(예시 형태)

- 구별 비교 코멘트:
    - “3구: 구상 핵심 이미지가 완성본에서 유지/약화/변형됨”
- 의미 보존 경고:
    - “대우/평측 맞추며 핵심 소재 교체됨(○○ → △△)”
- 한문 자연스러움 코멘트(주의):
    - 확정 판정이 아니라 **모호함/자연스러움 정도**로 표현(‘추천/진단’)

### 비용 절감 운영 원칙

- 호출 시점 제한: **저장 / 학인 / 감상하기 진입** 같은 이벤트에서만 호출
- 캐시: 동일 텍스트는 재호출하지 않음
- 배율 등 긴 작품은 련 단위 분할 호출 고려

---

## 3) 번역 기능: “그럴듯한(검증용) 한시 번역”은 AI로 가능

### 요구사항

- 사용자가 자신의 시가 “말이 되는지”, “구상과 맞는지” 확인하고 싶어 함.
- 번역은 예술 번역이 아니라 **의미 검증을 위한 번역**이 핵심.

### 결론

- 일반 번역 API는 문언문/한시에 취약할 수 있음.
- LLM(OpenAI/Claude/Gemini)이 **직역+의역+주석** 형태로 더 적합.
- 서비스에서는 번역을 “정답”이 아니라 **검증용 참고 레이어**로 제공.

## 4) 저장/공유/커뮤니티(자랑/댓글) – “DB 없는 프론트 단독” 전제에서의 현실 해법

### 배경

- 현재 사이트는 DB 없이 프론트엔드만으로 동작(정적/JSON 기반).
- 사용자는:
    - 자신의 작품 저장
    - 다른 사람에게 자랑(공유/갤러리)
    - 댓글/코멘트 기능
    을 원할 수 있음.
- 그러나 계정/정책이 없으면 **욕설·광고·도배** 문제가 즉시 발생.

---

## 5) DB 없이 가능한 기능(권장 1단계): 개인 저장 + 파일 백업 + 링크 공유

### 5.1 개인 저장(계정 없음)

- LocalStorage/IndexedDB에 작품 저장
- 장점: 서버/정책/비용 거의 0, 구현 쉬움
- 단점: 기기 변경/브라우저 데이터 삭제 시 소실

**권장 UI**

- “내 작품함(이 기기)” 탭 제공
- 목록/검색/태그/정렬

### 5.2 내보내기/가져오기(파일 기반 백업)

- “내 작품 내보내기” → `my_poems.json` 다운로드
- “가져오기” → 파일 업로드로 복원
- 장점: 프론트만으로 백업 가능
- 단점: 사용자 행위 필요

### 5.3 서버 없는 공유 링크(압축 URL)

- 작품 데이터를 URL(hash/query)에 **압축+인코딩**하여 공유 링크 생성
- 장점: 서버 없이 “공유” 가능
- 단점: URL 길이 제한/노출 위험/수정·삭제 개념 부재

> 결론: DB 없이 “자랑”을 구현한다면 **공유 링크 + 이미지(시화) 다운로드**가 최선.
> 

---

## 6) 댓글/갤러리는 프론트 단독으로 안전하게 불가 (정직한 결론)

- 계정/권한/수정·삭제/모더레이션이 없으면 공용 공간은 스팸·욕설·광고를 막기 어렵다.
- 따라서 “공용 갤러리/댓글”은 **최소한의 서버(또는 BaaS)**가 사실상 필요함.

---

## 7) 최소 추가로 커뮤니티를 여는 방법(권장 2단계): BaaS 또는 서버리스

### 옵션 A) Firebase / Supabase (BaaS, 정석)

- 저장: Firestore 또는 Postgres(Supabase)
- 인증: Google/Apple OAuth / 이메일 매직링크 / (옵션) 익명 로그인
- 권한: 보안 규칙으로 “본인 글만 수정/삭제” 강제 가능
- 모더레이션: 로그인 사용자만 작성 + 레이트리밋 + 신고/숨김 + (필요 시) AI 모더레이션

### 옵션 B) Cloudflare Workers + KV/D1 (가벼운 백엔드)

- 비용/속도 유리, 운영 부담 낮음
- 단, 설계/구현은 BaaS보다 손이 감

### 옵션 C) GitHub Issues/Discussions 활용(특수)

- 계정/스팸 방어를 GitHub에 위임 가능
- 일반 사용자 UX가 어색하여 보통은 부적합

---

## 8) “계정 없이 댓글”은 거의 불가능 → 현실적 타협안

### 추천 타협안

- **읽기(갤러리)는 누구나**, **쓰기(게시·댓글)는 로그인**
- 로그인은 “회원가입 폼”이 아니라:
    - Google/Apple 1클릭 또는 이메일 매직링크
- 초기에는 댓글 대신:
    - “좋아요/북마크/공유”만 제공(스팸 위험 낮음)

---

## 9) DB 없이도 ‘자랑 욕구’를 만족시키는 프론트-only 아이디어: 시화(詩畫) 만들기

- 작품을 캔버스/HTML 렌더링으로 **PNG로 다운로드**하게 만든다.
- 수묵화 템플릿 + 서예 느낌 폰트 + 낙관(도장) 스타일 등으로 “공유하기 좋은 결과물” 제공.
- 커뮤니티는 사이트 내부가 아니라 외부(SNS/커뮤니티)로 유도 가능.

> 결론: 당장 DB가 없다면 “갤러리/댓글” 대신 “시화 PNG + 링크 공유”로 자랑 흐름을 확보할 수 있음.
> 

---

## 10) 권장 로드맵(운영/리스크 최소화)

### 1단계(현재: 프론트-only 유지)

- 내 작품함(LocalStorage/IndexedDB)
- 내보내기/가져오기(JSON)
- 공유 링크(압축 URL, 가능 시)
- 시화 PNG 다운로드
- 갤러리/댓글은 보류

### 2단계(갤러리/댓글이 필요해졌을 때)

- Supabase 또는 Firebase 도입
- OAuth(구글/애플)로 최소 로그인
- 게시/삭제/신고/숨김 + 레이트리밋
- 댓글은 초기 비활성 또는 제한적 오픈(스팸 방어 확보 후 확대)

## 1) 대우/요체(拗體·拗救) 문제 제기와 해결 방향

### 1-1. 문제 인식

- 형식(절구/율시의 구수·자수·압운·평측) 검사는 엔진으로 처리 가능.
- 그러나 **대우(對偶)**의 적합성이나, 빨간불(율격 위반)을 **요체/요구로 살리는 결정**은 엔진만으로 해결하기 어렵다는 우려가 제기됨.

### 1-2. 결론(역할 분리)

- **엔진이 확정 판정**해야 하는 것:
    - 어디가 위반인지(좌표/근거)
    - 요(拗)로 인정 가능한지 여부
    - 요구(救)를 해야 할 슬롯(구제 지점) 산출
- **AI가 도울 수 있는 것(후보/설명/코칭)**:
    - 엔진이 내준 “구제 슬롯”에 대해 의미를 유지하면서 평측/자수 제약을 만족하는 **후보 시어/한자 추천**
    - 대우가 약한 경우, 대응 구를 **대우로 성립하도록 수정 후보**를 제약 하에서 생성
- 핵심 원칙: AI는 “판정자”가 아니라 “코치”이며 자동 적용은 지양하고 후보를 제시한다.

## 3) 8-1 ~ 8-4 추가 조건(사용자 측 결정) 정리

### 3-1. 8-1 대표 작품/저작권

- 대표 작품은 당시삼백수 DB에서 선정.
- 고전 원문 저작권 없음.
- 번역문은 공공누리 데이터이므로 **표시/출처 표기**로 처리 가능.

### 3-2. 8-2 평측/운부 기준 정책

- 정책 문제이므로 교수님과 협의 필요(서비스 기준 확정 전까지 모듈화 권장).

### 3-3. 8-3 요구 패턴 확장

- 장기적으로 고급 단계까지 확장 의지.
- 다만 무한 범위가 되므로 “지원할 요구 유형 리스트업” 기반으로 단계적 확장 필요.

### 3-4. 8-4 대우 품사 힌트 비용 이슈

- 실시간 모델 기반 품사/대우 힌트는 토큰 비용 부담이 큼.
- 사전 기반 힌트로 시작하고, AI 개입은 “완성/검사 단계의 종합 코멘트”로 제한하는 방향이 합리적.

## 4) 추가 요구: 구상(스토리보드) 대비 구현도 검증 + 문장 자연스러움

### 4-1. 문제

- 평측/대우 맞추느라 도치/생략/치환이 생기면,
    - 사용자가 “내 구상이 제대로 반영됐는지”,
    - “한문이 말이 되는지(자연스러운지)”
    를 확인하고 싶어함.

### 4-2. 결론

- 이 영역은 엔진으로 한계가 있고, **AI로 ‘검증 레이어’**를 제공하는 것이 적합.
- 실시간이 아니라 **저장/학인/감상** 같은 이벤트에서만 호출하여 비용을 통제.

### 4-3. 권장 출력(검증용 번역/해설)

- **직역(검증용)**: 생략된 요소는 대괄호로 보충
- **의역(읽기용)**: 자연스러운 한국어 풀이
- **주석(해설)**: 도치/생략/핵심 어구 설명

### 4-4. 구상 vs 완성 리포트

- 구별 의미 일치/약화/변형 코멘트
- 의미 보존 경고(핵심 소재 교체 등)
- 한문 자연스러움 코멘트(확정 판정이 아니라 진단/추천 톤)

---

## 5) 저장/공유/커뮤니티(자랑/댓글) 문제와 해결책

### 5-1. 제약

- 현재 사이트는 DB 없이 프론트 중심.
- 저장은 가능하지만 공용 갤러리/댓글은 스팸/욕설/광고 위험이 큼.

### 5-2. DB 없이 가능한 것(1단계 권장)

- 로컬 저장(LocalStorage/IndexedDB)
- 내보내기/가져오기(JSON 파일)
- 공유 링크(압축 URL) — 길이 제한/삭제 불가 등의 제약 존재
- 시화(詩畫) PNG 다운로드(프론트-only로도 가능)로 “자랑” 욕구를 외부 SNS 공유로 해결

### 5-3. 공용 갤러리/댓글을 원하면(2단계)

- 최소 서버 또는 BaaS(Firebase/Supabase) 필요:
    - 인증(OAuth/매직링크)
    - 권한(본인 글 수정/삭제)
    - 모더레이션(신고/숨김/레이트리밋, 필요 시 AI 모더레이션)

### 5-4. 현실적 타협안

- 읽기(갤러리)는 누구나, 쓰기(게시·댓글)는 로그인
- 초기에는 댓글 대신 좋아요/북마크 중심으로 운영 가능

---

## 6) AI 검증 기능 과금/제한 정책 논의

### 6-1. 검증 AI: 파인튜닝 필요성

- “검증 번역/구상 비교/대우 언급” 수준은 **파인튜닝 없이** 가능.
- 프롬프트 고정 + few-shot + 출력 제한으로 안정화 권장.

### 6-2. 토큰/비용 대략 추정(개념)

- 시 본문(칠언율시 56자 등)은 입력 토큰 규모가 크지 않음.
- 비용은 대체로 **고정 프롬프트 + 출력 길이**가 좌우.
- 운영 측면에서 출력 상한/호출 시점 제한/캐시가 중요.

### 6-3. 맛보기 정책

- “가입 시 AI 검증 3회 무료, 이후 구독” 정책이 가능하다는 결론.
- 단, 프론트-only 카운팅은 우회가 쉬우므로:
    - 최소한의 서버/BaaS 기반 카운팅 또는 익명 UID 방식 필요.
- API 키는 프론트에 두지 않고 서버리스에서 호출해야 함.
