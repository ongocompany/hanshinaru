기준 문서:

## 2. 엔진 모듈 설계

### 2.1 모듈 구성
1. `templateResolver`
- 시체/격식/입운 설정으로 목표 템플릿 확정

2. `meterValidator`
- 구수/자수 검사

3. `rhymeValidator`
- 압운 위치, 운부 일치, 환운/출운/락운 탐지

4. `toneValidator`
- 슬롯별 평측 위반 점검

5. `coupletChecker`
- 대우 구간(율시 3-4, 5-6) 구조/길이/기초 품사 힌트

6. `illnessDetector`
- 고평, 삼평미/삼측미 등 병 탐지(단계적 확장)

7. `yaojiuResolver`
- 요체 인정 가능 여부 + 구제 슬롯 계산

### 2.2 검사 실행 순서
1. 템플릿 확정
2. 자수/구수 불합격이면 치명 오류 우선 반환
3. 압운 검사
4. 평측 검사
5. 병 검사
6. 대우 구조 검사
7. 요체 처리 가능성 분석
8. 통합 리포트 생성

## 3. 실시간 모드 알고리즘

### 입력 단위
- 글자 입력 시 해당 구(라인)만 증분 검사

### 신호등 규칙
- 빨강: 구조 위반(자수 초과, 압운 슬롯 위반, 운부 불가)
- 노랑: 병 위험, 대우 균형 약함, 평측 불안
- 초록: 현재 상태 기준 형식 통과

### 성능 전략
- 디바운스(150~250ms)
- 라인 단위 캐시 키: `type + lineText + rhymeGroup + options`

## 4. AI 모듈 설계

### 4.1 AI 호출 지점
1. Step 2/3: 주제·소재 정리
2. Step 4: 운부/운자 분위기 추천
3. Step 6(B 모드): 특정 슬롯 후보 추천
4. Step 7: 검증 번역/구상 대비 리포트/대우 진단

### 4.2 AI 출력 계약(강제)
- 항상 JSON 반환
- 엔진 제약 전달 필수:
  - 글자수
  - 운부 제한
  - 평측 제한
  - 금지어/유지어
- 후보 개수 제한(예: 5개)

### 4.3 안전장치
1. AI 후보 재검증
- AI가 준 후보를 엔진으로 재검사 후 통과분만 노출

2. 과잉개입 방지
- “완성 시 전체 자동 재작성” 기능 금지
- 후보 선택은 사용자 수동 선택

3. 비용 통제
- 호출 이벤트 제한(학인/감상/명시 버튼)
- 동일 입력 캐시

## 5. 대우(對偶) 하이브리드 로직

### 엔진 책임
- 대우 구간 존재/자수 대응
- 기초 품사 힌트(사전 기반)
- 반복/불균형 경고 좌표

### AI 책임
- 의미 범주 대응 진단
- 대비 강화/문맥 유지 대안 제시
- 제약 충족 후보 생성

### 출력 구조
```json
{
  "engineCouplet": {
    "linePair": [3,4],
    "checks": ["LENGTH_OK", "SLOT_OK", "POS_HINT_WARN"],
    "issues": [{"slot": 2, "message": "품사 대응 약함"}]
  },
  "aiCoupletCoach": {
    "summary": "시간-공간 대응은 좋지만 동사 대응이 약함",
    "candidates": ["...", "...", "..."]
  }
}
```
