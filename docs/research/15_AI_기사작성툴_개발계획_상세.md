# AI 기사작성툴 개발 계획서 (상세)

본 문서는 `docs/research/14_AI_기사작성툴_제작계획_260222_GPT.md`와 `docs/FromJin/12_남은사항들_작업계획서_CL.md`의 내용을 바탕으로 재구성한 상세 개발 계획서입니다. 특히 사용할 AI 모델과 전체 파이프라인의 흐름을 구체화하는 데 중점을 두었습니다.

---

## 1. 프로젝트 개요 및 목표

- **목적**: 메인 페이지의 '한시관련 소식' 영역(현재 하드코딩된 플레이스홀더)을 자동화된 뉴스 크롤링 및 AI 재작성 시스템으로 전환.
- **주요 특징**: 뉴스를 단순 스크랩하는 것이 아니라 **AI를 통해 필터링하고 저작권 문제가 없는 300~500자 분량의 새로운 기사로 재작성(Rewrite)** 하여 제공.
- **예상 볼륨**: 주 1회 수집, 월 8~12건의 양질의 기사 생성.
- **노출 방식**: 메인 페이지 아코디언 UI (최신 5~8건 노출). 별도 게시판 운영 안 함.

---

## 2. 권장 AI 모델 선정

기사 필터링 및 재작성을 수행할 AI 모델로는 **Claude 3.5 Sonnet** 또는 **GPT-4o/GPT-4o-mini**를 권장합니다.

### 모델 비교 및 추천

| 모델 | 추천도 | 장점 | 단점 | 활용 방안 |
| :--- | :---: | :--- | :--- | :--- |
| **Claude 3.5 Sonnet** | **매우 높음** | 문학적이고 차분한 한국어 톤앤매너 작성에 탁월함. 한시 및 전통문화 관련 문맥(Nuance) 이해도가 높음. | GPT-4o-mini 대비 API 단가가 다소 높음. | 메인 기사 재작성용 (본문 생성) |
| **GPT-4o-mini** | **높음** | 가성비가 매우 뛰어나며, 처리 속도가 빠름. | 깊이 있는 문학적 어조 구현에서 Claude 대비 약간 건조할 수 있음. | 대량의 기사 1차 필터링용 (관련성 판별) |

### 최종 선정 전략: 하이브리드 또는 단일 모델
- **단일 모델 사용 시 (추천)**: 비용이 크게 부담되지 않는 볼륨(월 10여 건)이므로 결과물의 품질(문체, 교양 있는 어조)을 위해 **Claude 3.5 Sonnet** 단독 사용을 가장 추천합니다.
- **하이브리드 사용 시**: 크롤링된 대량의 기초 데이터(수십 건)에서 "한시/전통문화와 진짜 관련이 있는가?"를 판별하는 1차 필터링은 저렴한 **GPT-4o-mini**로 수행하고, 최종 채택된 2~3건의 뉴스 본문 재작성만 **Claude 3.5 Sonnet**으로 수행하여 비용과 품질을 모두 최적화할 수 있습니다.

## 3. 전체 파이프라인 흐름 (Data Flow Pipeline)

파이프라인은 크게 **뉴스 수집 → 정제/필터링 → AI 재작성 → 저장 및 배포**의 4단계로 구성됩니다. 이 모든 과정은 **GitHub Actions**를 통해 매주 월요일 오전 9시(UTC 기준)에 자동 실행되도록 구성합니다. (`.github/workflows/crawl-news.yml`)

### Step 1: 뉴스 소스 수집 (Crawling)
- **실행 스크립트**: `scripts/crawl_news.js`
- **데이터 소스**:
  1. **Naver 검색 API (주력)**: 뉴스 검색 API 활용 (무료 일 25,000건 한도). 최적화된 검색 파라미터(정확도순/최신순) 사용.
  2. **Google News RSS (보조)**: 네이버에서 놓치는 외신이나 전문 매체의 기사 보완.
- **검색 키워드 (6개 카테고리)**:
  - 漢詩/한시: 한시, 당시, 고전시, 시선집, 당시삼백수
  - 교육/시험: 한자능력검정시험, 한자교육, 한문교육
  - 시인 화제: 두보, 이백, 왕유, 백거이 등 + 뉴스/전시/발견
  - 시문학: 신춘문예 시문학, 문학상 시부문
  - 전통행사: 석전대제, 성균관, 한시 백일장, 유교문화 행사
  - 서예/전시: 서예전, 한시 전시회, 서각

### Step 2: 중복 제거 및 AI 1차 필터링
- **중복 제거**: 기존에 저장된 `public/index/news_articles.json`의 원문 URL 및 제목과 대조해 이미 처리한 기사는 제외 (URL HashSet 활용).
- **AI 1차 필터링 (Relevance Check)**:
  - 수집된 기사의 제목과 요약문을 AI에게 전달하여, **"이 기사가 한시, 전통문화, 한자 교육과 실질적인 관련이 있는 유의미한 뉴스인가?"** 판별 (동음이의어 기사, 스팸성 홍보 기사 필터링).
  - 프롬프트 예시: *"다음 뉴스 요약이 전통 한시 생태계나 한자 문화에 관련된 뉴스인지 판별하고 boolean으로 반환하라."*

### Step 3: AI 기사 재작성 (Rewrite)
1차 필터링을 통과한 유효 기사(주당 2~3건 예상)의 원문 본문(또는 상세 요약)을 스크래핑한 뒤 AI를 통해 재작성합니다.
- **지시 프롬프트 규율 (System Prompt)**:
  - 분량: 한국어 기준 300~500자 내외.
  - 형식: [기사 제목]과 [본문 1~2문단] 구조 유지. 하단에 원문 출처 표기 필수.
  - 톤앤매너: 방문자(교양 있는 문학 애호가) 수준에 맞게 차분하고 정제된 문체 사용.
  - 저작권 회피: 원문 문장을 그대로 복사하지 말고, 사실(Fact)만 추출하여 독자적인 서술 구조로 재구성할 것.

### Step 4: JSON 데이터 저장 및 배포
- **JSON 업데이트**: AI가 재작성한 결과물을 `public/index/news_articles.json` 파일에 병합합니다. 최신순으로 정렬하며, 생성일(`publishedAt`, `crawledAt`), 출처 URL(`sourceUrl`), 식별 썸네일/태그 정보 등을 구조화하여 저장.
- **보존 조건**: 데이터 비대화를 막기 위해 JSON 파일에는 최신 30~50건만 보존. 클라이언트(브라우저)에는 가장 최신 글 5~8건만 노출. 노출 2개월이 지난 기사는 사이트에서 숨기기.
- **자동 커밋/푸시**: 완료 후 GitHub Actions 워크플로우가 수정된 JSON 파일을 깃허브 저장소에 자동 `commit` & `push`.

### Step 5: 프론트엔드 렌더링
- **index.html**: 변경된 `news_articles.json`을 `fetch()`하여 메인 페이지의 아코디언 UI 영역에 동적 렌더링.

## 4. 모듈 개발 순서 (Phase)

1. **Phase 1 (환경구축 및 수집)**:
   - Naver API 및 Google News RSS 파서 로직 (`scripts/crawl_news.js` 초기 버전) 작성. 수집 데이터 정규화 및 콘솔 출력 데모.
2. **Phase 2 (AI 연동)**:
   - AI API (Claude 또는 OpenAI) 통신 로직 추가. 필터링 및 재작성 프롬프트 튜닝. JSON 구조 생성 및 저장 로직 구현.
3. **Phase 3 (프론트엔드)**:
   - 메인 `index.html` 하단에 가짜 플레이스홀더를 제거하고, `app.js`에 `news_articles.json`을 읽어와 아코디언 UI로 렌더링하는 함수 구현.
4. **Phase 4 (자동화)**:
   - `.github/workflows/crawl-news.yml` 파일 작성. 필요한 Secret 변수(API 키 등) 설정. 수동/스케줄 작동 테스트.

---
**요약**: 비용 소모가 거의 없고 품질 좋은 한시 소식을 매주 자동으로 사용자에게 제공할 수 있습니다. 기사의 품격을 위해 "Claude 3.5 Sonnet" 엔진을 채택하고, GitHub Actions의 cron 기능을 이용해 주 단위 크롤링-재작성-저장 워크플로우를 완성하는 것이 이 계획의 핵심입니다.
