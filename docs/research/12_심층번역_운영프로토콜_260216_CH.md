## 3. UI/렌더링 연동 규칙 (app.js/admin.js 기반)

### 3-1. 주석 표시 우선순위
- 화면 렌더링은 `notesOwned` 우선, 없을 때만 legacy `notes` 사용
- 즉, `notesOwned`가 있으면 기존 주석은 화면에서 사실상 숨김(미사용) 상태

### 3-2. 본문/집평 인라인 주석 방식
- 렌더러는 `[번호]` 마커와 `head` 매칭으로 인라인 주석을 붙인다.
- 현재 전처리 로직이 `head` 기반으로 마커를 자동 삽입하므로, `notesOwned.head`는 원문 어구와 최대한 동일하게 써야 한다.

### 3-3. 번역 표출 우선순위
- 시 번역: `translationKoOwned` 우선
- 집평 번역: `jipyeongKoOwned` 우선
- 집평 번역에서는 `[숫자]`, `역주` 표기 자동 클린업이 걸리지만, 작성 단계부터 넣지 않는 것을 원칙으로 한다.

## 4. 검색 조건 (심층조사 규칙)

### 4-1. 작품당 최소 출처 수
- 최소 3개 URL 필수 (`sourceRefs.length >= 3`)

### 4-2. 권장 출처 조합
1. 원문 기준본: CyberSeodang (`sourceUrl` 또는 `titleId` 기반 URL)
2. 원문 교차본: Wikisource (직접 문서 또는 Search URL)
3. 보조 해설/주석: 古文島/古詩文網/동급 공개 고전 해설 사이트

### 4-3. 전고/어휘 확인이 필요한 경우
- 인명·지명·전고·고문 어법이 핵심이면 보조 사전/고전 DB를 추가 조회
- 확인한 핵심은 `notesOwned`에 반영

### 4-4. 검색 쿼리 템플릿
- `시인명 + 작품명(한자)`
- `작품명 + 集評`
- `작품명 + 핵심구절(전고 의심 구)`
- `작품명 + 異文` 또는 `一作`

### 4-5. sourceRefs 기록 규칙
- 실제 확인한 URL만 기록
- 가짜 URL/미열람 URL 기록 금지
- 문서가 없으면 Search URL로 대체 가능하되, 최소 1개는 실제 본문 URL을 넣는다.

## 5. 작성 방법 (핵심)

### 5-1. 시 번역 (`translationKoOwned`)
- 기본: 직역+의역 균형
- 톤: 자연스러운 시어체
- 종결 어미: 단조로운 `~다` 반복 지양 (예: `~노라`, `~도다`, `~하노니`, `~하리` 등 상황 맞게)
- 행 수: 원문 행 구성 최대한 유지
- 금지: 기존 상용 번역 문장 복제

### 5-2. 집평 번역 (`jipyeongKoOwned`)
- 기본 원칙: **직역 우선**
- 문체: 현대어 평서문 (과도한 고어체 금지)
- 판단 어조/논리 전개/구절 인용 순서를 임의로 바꾸지 않는다.
- `역주`, `[숫자]`, `[역주숫자]` 표기 제거
- `jipyeongZh`가 비어 있으면 아래 고정 문장 사용:
  - `해당 수록본에는 별도의 집평 원문이 실려 있지 않다.`

### 5-3. 주석 (`notesOwned`)
- 작품당 5~8개 권장 (최소 5)
- 우선 선정 항목:
  - 전고
  - 고유명사(인명/지명/관직/서명)
  - 현대어와 의미 차가 큰 어휘
  - 도치/함축/요체 구문
  - 이문(一作) 정보
- `notes`(기존 주석)는 사실 체크리스트로만 참고하고 문장 복제 금지
- `head`는 가능한 원문에 그대로 있는 표기로 작성 (인라인 매칭 안정성)
