## 1. 태훈이 리서치에 대한 총평

### 07_hanshi.kr 종합보고서 — 👍 잘 뽑았음
- 사이트 구조 파악, 접속자 통계, 평측사전 기능 분석 모두 깔끔함
- 특히 **접속 회원 630명 / 일일 200명대 / 누적 52만**이라는 수치가 의미있음 → 한시 작법 도구에 대한 **실수요가 검증**된 셈
- 한 가지 아쉬운 점: hanshi.kr의 평측사전이 구체적으로 **어떤 운서(평수운? 광운?)** 기준인지까지는 파악 못 한 듯. 근데 이건 로그인해서 직접 돌려봐야 아는 거라 브라우저 차단 상황에서는 어쩔 수 없었을 거임

### 08_평측알고리즘 타당성검토 — ⚠️ 방향은 맞지만 보완 필요
- **"JS로 100% 구현 가능"이라는 결론 자체는 정확함.** 평측 패턴 매칭은 본질적으로 배열 비교 문제가 맞음
- 그런데 수도코드 예시에 **치명적 오류가 하나 있음:**

```diff
  // 봉요(蜂腰) — 태훈이 코드
- if (lineTones[1] === lineTones[3]) {
-     return "경고: 봉요(蜂腰) 발생!";
- }
```

이건 **봉요의 정의를 잘못 이해**한 것 같음. 나무위키와 학술 자료를 보면:

- **봉요(蜂腰)**: 5언에서 **제2자와 제5자**의 평측이 같을 때 (혀리가 잘록한 벌허리 모양). 출처마다 약간 차이가 있는데, 심약(沈約)의 원래 정의는 "제2자와 제5자의 성조가 같은 것"임
- 태훈이가 적은 "2자와 4자 비교"는 봉요가 아니라 일반적인 **점/점 위치 체크**에 더 가까움

이런 부분이 바로 **형이 Phase 2 완료 시점에서 검수해야 할 핵심 포인트**임. 알고리즘의 뼈대는 단순하지만, **각 병의 정확한 정의**를 학술적으로 확정하는 게 더 어려운 문제.

### 09_팔병/요구 UX 아이디어 — 🔥 이건 진짜 좋음! 형 아이디어 포함

4가지 아이디어 모두 채택 가치가 있음. 특히:

| 아이디어 | 민철 평가 | 비고 |
|---------|---------|------|
| 1. Error vs Warning 분리 | ✅ 채택 | 내 설계서의 "신호등 3색"과 정확히 일치. 형의 DB 연동 팝업은 **킬러 피처** |
| 2. 요구 Gamification | ✅ 채택 (조건부) | 빨강→금색 전환 애니메이션은 기술적으로 가능하나, **요구 패턴 판정 자체가 Phase 2-6 (★★★)**. 애니메이션은 Phase 3 |
| 3. Strict Mode 토글 | ✅ 채택 | 내 로드맵 Phase 0-3(수익화 모델)과 직결. 초보/백일장 모드는 **프리셋 구현의 핵심** |
| 4. Ignore 기능 | ✅ 채택 | 린터 suppress 패턴. 상태 관리 JSON에 `ignored: [셀좌표]` 배열만 추가하면 됨 |

## 2. 나무위키 기준 팔병(八病) 전수 조사 & 구현 난이도

나무위키에서 정리한 심약(沈約)의 팔병 8가지를 기준으로 구현 난이도와 우선순위를 정리함:

| # | 병명 | 한자 | 정의 | 검사 데이터 | 구현 난이도 | 우선순위 |
|---|------|------|------|-----------|-----------|---------|
| 1 | **평두(平頭)** | 平頭 | 1련의 출구와 대구에서 제1·2자의 평측이 같음 | 평측 배열 | ★ | P1 |
| 2 | **상미(上尾)** | 上尾 | 1련의 출구와 대구에서 마지막 글자의 성조가 같음 (압운 무시) | 평측 배열 | ★ | P1 |
| 3 | **봉요(蜂腰)** | 蜂腰 | 5언에서 2자와 5자의 평측이 같음 | 평측 배열 | ★ | P1 |
| 4 | **학슬(鶴膝)** | 鶴膝 | 제1련 5자와 제2련 5자의 평측이 같음 | 평측 배열 (련간 비교) | ★★ | P1 |
| 5 | **대운(大韻)** | 大韻 | 출구에 운자(압운자)와 같은 운부의 글자가 올라옴 | 평측 + **운부(rhyme_key)** | ★★ | P2 |
| 6 | **소운(小韻)** | 小韻 | 인접한 두 구에서 같은 운부 글자가 반복 | 운부(rhyme_key) | ★★ | P2 |
| 7 | **방뉴(傍紐)** | 傍紐 | 같은 구 안에 **쌍성(声母가 같은)** 글자가 있음 | **성모(초성) 데이터 필요** | ★★★★ | P3 |
| 8 | **정뉴(正紐)** | 正紐 | 같은 구 안에 **첩운(韻母도 같은)** 글자가 있음 | **성모 + 운모 데이터 필요** | ★★★★ | P3 |

### 핵심 발견: 팔병 1~6 vs 7~8은 차원이 다름

- **1~6번**: `char_map v2`의 평측(tone) + 운부(rhyme_key) 데이터만으로 **100% 구현 가능**
- **7~8번 (방뉴/정뉴)**: 성모(聲母, 초성) 데이터가 추가로 필요함. 현재 우리의 `char_map`에는 이 정보가 없음. 이걸 추가하려면 **병음(pinyin) 파싱** 또는 별도 성모 DB 구축이 필요

> 💡 **내 제안**: 방뉴/정뉴는 사실 당대 이후로 엄격하게 적용되지 않았고, 
> 현대 한시인들도 거의 신경 쓰지 않음. **MVP에서는 P3(장기)로 빼놓고, 
> 팔병 1~6만 먼저 구현**하는 게 맞음.

## 3. 형 아이디어("DB 시 예시 팝업")에 대한 구현 로드맵 추가

형이 제안한 "노란불 뜰 때 우리 DB의 명시 예시를 보여주는 UX"를 구현하려면 **선행 작업**이 하나 필요함:

### 사전 작업: poems.full.json 팔병/요구 태깅

현재 `poems.full.json`의 320편에 대해 팔병 엔진을 돌려서 **각 시에 어떤 병이 있는지 사전에 태깅**해둬야 함.

```json
// poems.full.json 확장 예시
{
    "title": "靜夜思",
    "author": "李白",
    "category": "五言絶句",
    "content": "床前明月光...",
    "illnesses": ["봉요_L1", "삼평미_L4"],  // ← 사전 태깅
    "rescues": ["자구_L3"]                    // ← 요구 패턴 태깅
}
```

이 태깅은 **Phase 2 엔진 완성 직후**에 스크립트로 일괄 처리 가능. Phase 2-8(신호등 판정 통합) 태스크 완료 → 바로 320편 일괄 스캔 → 태깅 결과 저장.

## 5. 업무 분담 제안

### 민철 (즉시 착수 가능)
1. Phase 1-1~1-2: `char_map v2` 확장 (다음자 배열화 + 입성 toneDetail)
2. Phase 2-4~2-5: 팔병 1~6 탐지 엔진 구현 (정확한 정의 확인 후)
3. Phase 2-9 [NEW]: 320편 일괄 팔병/요구 태깅 스크립트

### 태훈
1. 팔병 8가지의 **학술적 정의 정밀 조사** — 특히 봉요의 "2·5자 vs 2·4자" 같은 출처간 차이 정리
2. 우리 DB(poems.full.json) 320편 중 **팔병/요구 사례가 있는 유명 시** 수작업 샘플링 (5~10편)
   - 이건 형의 한시 지식이 필수임. 엔진으로 기계적 태깅하기 전에 형이 직접 "이 시에서 이런 변격을 씀"이라고 확인해줘야 검증 기준이 됨
3. hanshi.kr 평측사전과의 **결과 비교 데이터** 수집 (형이 로그인해서 특정 시를 돌려보고, 결과 화면을 캡처해서 공유)

### 형
- Phase 2 완료 시 **실제 한시 예제로 팔병 검사 결과 검수**
- DB 시 예시 팝업의 **설명 문구 감수** ("이백도 여기서 봉요를 신경쓰지 않고..." 같은 해설)
- 필요시 hanshi.kr 평측사전 결과 캡처 공유
