# AI 번역·집필 파이프라인 설계

작성일: 2026-02-16  
작성자: CH(지훈)  
대상 프로젝트: Tangshi

## 1. 설계 목적
본 설계는 서비스 출시를 전제로, 라이선스가 있는 기존 번역문을 재사용하지 않고 자체 콘텐츠를 안정적으로 생산하기 위한 작업 표준을 정의한다.  
대상 텍스트는 시 본문(한문 원문)과 한자 집평 원문으로 한정한다.  
집평 번역문(기존 한국어 번역)은 작업 입력에서 제외한다.

## 2. 입력 범위와 금지 범위
### 2-1. 허용 입력
- `poemZh` (시 본문 한문 원문)
- `jipyeongZh` (집평 한문 원문)
- 메타데이터: 시인, 작품번호, 권차, 장르, 운율 정보 등 비번역 필드

### 2-2. 금지 입력(모델 주입 기준)
- `translationKo` (기존 시 번역문)
- `jipyeongKo` (기존 집평 번역문)
- 제3자 라이선스 번역문 전문

### 2-3. 운영 원칙
생성 모델 컨텍스트에는 허용 입력만 주입한다.  
금지 입력 필드는 API 레벨에서 기본 차단한다.  
다만 편집자의 표절/어색함 판단을 위해 `translationKo`, `jipyeongKo`는 읽기 전용 비교 패널에서만 노출한다.  
읽기 전용 비교 패널의 텍스트는 모델 프롬프트로 전달하지 않는다.

## 3. 산출물 정의
본 파이프라인은 작품 1건당 다음 산출물을 만든다.

1. 시 본문 신규 번역
- 직해 중심 1안
- 문학적 완성도를 높인 1안

2. 한자 집평 신규 번역
- 원문 정보 구조를 유지한 해설 번역
- 핵심 평가 문장 요약

3. 보조 집필 데이터
- 주석 필요 구간
- 전고/용어 해설
- 참고 출처 목록과 인용 근거

## 4. 전체 파이프라인
### 단계 A. 데이터 준비
원문 텍스트를 행 단위로 분절하고, 주석 마커를 분리한다.  
작품 단위 식별자(`poemNoStr`, `titleId`)를 부여한다.

### 단계 B. 검색 계획 생성
원문 기반으로 검색 질의를 다국어로 생성한다.  
검색 언어는 중국어 간체/번체, 한국어, 영어를 기본으로 사용한다.

### 단계 C. 외부 검색 및 근거 수집
신뢰 가능한 1차/2차 자료를 우선 수집한다.  
근거 문장은 출처 URL, 문단 위치, 추출 시각과 함께 저장한다.

### 단계 D. AI 초안 생성
근거 요약을 입력으로 시 번역, 집평 번역, 주석 초안을 생성한다.  
이때 모델은 특정 출판 번역 문체를 모사하지 않도록 제한한다.

### 단계 E. 라이선스 리스크 점검
출력문을 금지 데이터 및 공개 번역 데이터와 유사도 검사한다.  
임계치 초과 구간은 자동 표시하여 재작성 대상으로 넘긴다.

### 단계 F. 인간 집필/편집
편집자가 문장 구조, 어휘 선택, 해설 논리를 재집필한다.  
수정 이력은 버전 단위로 저장하고 최종 승인자를 기록한다.

### 단계 G. 배포 패키징
최종 집필 결과는 기존 배포 경로인 `public/index/poems.full.json`에 병합 저장한다.  
기존 라이선스 필드(`translationKo`, `jipyeongKo`)는 보존하고, 신규 자체 집필 필드를 추가해 배포한다.

## 5. AI 검색 로직 설계
### 5-1. 질의 생성 규칙
질의는 3개 층으로 생성한다.

1. 원문 직접 질의
- 예: `杜甫 春望 國破山河在 注釋`
- 예: `感時花濺淚 恨別鳥驚心 解讀`

2. 작품+배경 질의
- 예: `杜甫 春望 安史之乱 写作背景`
- 예: `Du Fu Spring View historical context`

3. 집평 원문 질의
- 예: `司馬光 續詩話 國破山河在 城春草木深`
- 예: `杜詩詳註 春望`

### 5-2. 소스 우선순위
소스는 등급별로 가중치를 부여한다.

1. A등급(최우선)
- 고전 원문 아카이브(문헌 원문, 주석서 원문)
- 대학/공공기관 디지털 고문헌

2. B등급
- 학술기관 해설, 전문 백과, 대학 강의 노트

3. C등급
- 일반 블로그/커뮤니티 요약 자료
- C등급은 A/B의 보조 검증 용도로만 사용한다.

### 5-3. 근거 점수화
각 근거는 다음 점수를 합산한다.
- 출처 신뢰도 점수
- 원문 직접성 점수(2차 요약보다 원문 인용이 높음)
- 문맥 일치 점수(해당 구절 직접 설명 여부)
- 상충 여부 감점(타 자료와 충돌 시 감점)

### 5-4. 충돌 해결 규칙
서로 다른 해석이 충돌하면 다수결을 쓰지 않는다.  
원문 직독 가능성과 고주석 계보를 우선한다.  
결론이 하나로 수렴되지 않으면 복수 해석을 병기한다.

### 5-5. 저장 단위
검색 결과는 작품별 `evidence bundle`로 저장한다.
- 질의 문자열
- 수집 시각
- 출처 URL
- 근거 텍스트 스니펫
- 신뢰도 점수
- 적용된 번역/주석 문장 ID

## 6. 생성 프롬프트 정책
프롬프트에는 다음 제약을 고정한다.

1. 입력 제약
- 제공된 원문과 검색 근거 외 텍스트를 인용하지 않는다.

2. 문체 제약
- 문체는 현대 한국어 평서문을 기본으로 한다.
- 직역과 의역을 분리해 제시한다.

3. 저작권 제약
- 기존 번역의 표현을 복제하지 않는다.
- 특정 출판본과 유사한 문장 패턴을 피한다.

4. 출력 구조
- 번역
- 해설
- 주석 포인트
- 전고 정리
- 참고 출처

## 7. 유사도 및 품질 게이트
### 7-1. 유사도 검사
문장 단위 n-gram + 임베딩 유사도로 이중 검사한다.  
임계치 예시는 다음과 같다.
- 8-gram 중복률 15% 초과: 경고
- 임베딩 코사인 0.92 초과: 재작성 권고

### 7-2. 품질 검사
다음 조건을 통과해야 승인한다.
- 원문 행 누락 없음
- 핵심 어휘 오역 없음
- 역사 배경 연도/인명 오류 없음
- 출처 URL 누락 없음

### 7-3. 인간 승인 게이트
최종 승인 전에는 반드시 사람이 문장을 직접 수정한다.  
단순 맞춤법 수정만으로 승인하지 않고, 해석 논리와 문장 리듬을 재구성한다.

## 8. 내부 웹에디터 설계
### 8-1. 목표
편집자가 원문과 번역문을 동시 노출 상태로 비교하며 빠르게 재집필할 수 있어야 한다.  
표절 여부와 문장 어색함 판단을 위해 기존 번역문도 읽기 전용으로 즉시 대조 가능해야 한다.

### 8-2. 화면 구조(3패널)
1. 좌측 패널: 원문 패널
- 시 본문/집평 원문
- 행 번호
- 어휘 클릭 시 사전/용례 팝업

2. 중앙 패널: 집필 패널
- 원문-신규 번역 2열 동시 표시(행 단위 매핑)
- 번역 초안 편집
- 해설/주석/전고 섹션 편집
- 문장 블록 단위 버전 비교

3. 우측 패널: 근거 패널
- 검색 근거 목록
- 출처 클릭 시 원문 위치 하이라이트
- 근거-문장 연결(어느 문장을 뒷받침하는지 표시)
- 탭 2개 제공: `근거` / `레거시 번역 비교(읽기 전용)`

### 8-3. 편집 기능
- 블록 단위 저장(`Ctrl/Cmd+S`)
- 원문-신규 번역 동시 스크롤/행 동기화
- 직전 버전 되돌리기
- 문장 Diff 보기
- 유사도 경고 배지
- 출처 미기재 문장 경고
- 레거시 번역과의 고유사 문장 자동 하이라이트(검토용)

### 8-4. 작업 상태 모델
각 작품은 다음 상태를 가진다.
- `drafted` (AI 초안 생성 완료)
- `editing` (인간 편집 중)
- `reviewed` (검수 완료)
- `approved` (배포 승인)
- `published` (앱 반영 완료)

### 8-5. 데이터 구조(초안)
```json
{
  "poemId": "008",
  "inputs": {
    "poemZh": "...",
    "jipyeongZh": "..."
  },
  "legacyReadonly": {
    "translationKo": "...",
    "jipyeongKo": "..."
  },
  "drafts": {
    "translationLiteralKo": "...",
    "translationLiteraryKo": "...",
    "jipyeongKoDraft": "...",
    "notesKo": []
  },
  "publishTarget": {
    "file": "public/index/poems.full.json",
    "fields": {
      "translationKoOwned": "...",
      "jipyeongKoOwned": "..."
    }
  },
  "evidence": [],
  "similarity": {
    "status": "pass",
    "flags": []
  },
  "workflow": {
    "status": "editing",
    "editedBy": "JIN",
    "approvedBy": null
  }
}
```

### 8-6. `poems.full.json` 저장 규칙
배포 시 작품 오브젝트에 기존 필드는 유지하고 신규 필드를 추가한다.

1. 보존 필드
- `translationKo` (기존 라이선스 번역)
- `jipyeongKo` (기존 라이선스 집평 번역)

2. 신규 필드(자체 집필 결과)
- `translationKoOwned` (신규 시 번역 최종본)
- `jipyeongKoOwned` (신규 집평 번역 최종본)
- `ownedContentMeta` (작성/검수/승인 이력 메타)

3. 앱 노출 우선순위
- 시 번역: `translationKoOwned` 우선, 없으면 기존 표시 정책 fallback
- 집평 번역: `jipyeongKoOwned` 우선, 없으면 기존 표시 정책 fallback

4. 핵심 원칙
- 레거시 필드는 삭제/덮어쓰기 금지
- 신규 필드만 갱신
- 배포 전 JSON 스키마 검증 필수

## 9. 구현 로드맵
### 1단계(1주)
원문 추출기와 검색 수집기 프로토타입을 만든다.  
작품 10수로 샘플 파이프라인을 검증한다.

### 2단계(1~2주)
AI 초안 생성기와 유사도 검사기를 연결한다.  
품질 게이트와 로그 저장을 고정한다.

### 3단계(2주)
웹에디터 MVP를 구현한다.  
작품 단위 상태 전환, 버전 관리, 출처 연결 기능을 넣는다.

### 4단계(지속)
형님의 집필 스타일을 반영한 가이드라인을 추가한다.  
검수 기준표를 고도화하고 오류 패턴 사전을 축적한다.

## 10. 운영 체크리스트
배포 전 다음 항목을 확인한다.

1. 금지 입력 필드가 모델 컨텍스트에 주입되지 않았는가.  
2. 원문과 신규 번역이 편집 화면에서 동시 노출되어 행 단위 대조가 가능한가.  
3. 번역/해설의 핵심 문장이 외부 번역문과 과도하게 유사하지 않은가.  
4. 모든 해석 포인트에 최소 1개 이상의 출처 근거가 연결되어 있는가.  
5. 최종 문장에 편집자 실질 수정 이력이 남아 있는가.  
6. `public/index/poems.full.json`에 레거시 필드 보존 + 신규 필드 저장이 정상 반영됐는가.  
7. 결과 JSON에 승인자와 승인 시각이 기록되어 있는가.

## 11. 결론
출시 관점에서 가장 중요한 요소는 생성 자동화가 아니라 편집 통제권이다.  
이 설계는 AI를 초안 생산기로 제한하고, 사람의 재집필과 검증을 중심축으로 배치한다.  
따라서 라이선스 리스크를 낮추면서도 프로젝트 고유 문체와 해석 품질을 축적할 수 있다.
