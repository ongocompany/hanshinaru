# AI 기사작성툴 제작계획 (정리본)

## 1) 문서 목적
메인페이지 `한시관련 소식` 영역을 수동 플레이스홀더에서 자동 운영 체계로 전환하기 위한 계획을 정리한다.  
핵심은 `뉴스 수집 → AI 필터링/재작성 → JSON 저장 → 프론트 렌더링 → 주기 자동화`이다.

## 3) 확정된 기획 요약
- 대상: 메인페이지 "한시관련 소식" 섹션
- 운영 방식: 주 1회 자동 크롤링 + AI 기사 재작성
- UI: 별도 게시판이 아닌 메인 아코디언 UI
- 데이터 볼륨 가정: 월 8~12건
- 노출 정책: 최신 5~8건 노출, 오래된 기사(2개월+)는 화면 제외

## 4) 목표 아키텍처
1. 뉴스 소스 수집
- Naver 검색 API
- Google News RSS

2. 처리 파이프라인
- 키워드 검색
- 중복 제거
- AI 관련성 필터링(한시/한자/전통문화 연관성)
- AI 기사 재작성
- JSON 저장

3. 배포/자동화
- GitHub Actions 주기 실행
- 결과를 `public/index/news_articles.json`에 반영
- 자동 커밋/푸시

## 5) 키워드 전략 (요약)
- 한시/漢詩: 한시, 당시, 고전시, 漢詩, 시선집, 당시삼백수
- 한자 교육/시험: 한자능력검정시험, 한자교육, 한문교육 등
- 유명 시인 화제: 두보, 이백, 왕유, 백거이, 이상은 + 뉴스/전시/화제
- 한국 시문학: 신춘문예, 문학상 시부문, 시집 출간
- 전통문화 행사: 석전대제, 향교, 한시 백일장, 유교문화 행사
- 서예/전시: 서예전, 서예 공모전, 한시 전시회

## 6) AI 기사 재작성 규칙
- 분량: 300~500자(한국어)
- 형식: 기사 제목 + 본문 + 원문 출처 링크
- 톤: 교양 있고 차분한 문체, 시/한문 문화 친화적
- 저작권 원칙: 단순 요약이 아닌 사실 기반 재작성
- 모델: Claude 또는 GPT (건당 저비용)
- 품질 필터: 무관 기사 제거 후 채택

## 7) 데이터 구조 (요약)
`public/index/news_articles.json`

필드 예시:
- `id`
- `category`
- `title`
- `body` (300~500자 재작성 본문)
- `source`
- `sourceUrl`
- `publishedAt`
- `crawledAt`
- `keywords`

## 8) 자동화 워크플로우
- 파일: `.github/workflows/crawl-news.yml`
- 트리거: 주 1회 cron + 수동 실행
- 실행 단계:
1. 의존성 설치
2. `node scripts/crawl_news.js` 실행
3. `public/index/news_articles.json` 갱신
4. 자동 커밋/푸시

## 9) 파일/구현 단위
- `scripts/crawl_news.js`: 크롤링 + AI 재작성 스크립트
- `public/index/news_articles.json`: 기사 데이터 저장소
- `.github/workflows/crawl-news.yml`: 주기 자동화
- `index.html`: 메인 뉴스 섹션 아코디언 렌더링

## 10) 단계별 구현 계획
- Phase 1: 크롤링 스크립트(Naver API + 키워드 검색)
- Phase 2: AI 필터링 + 기사 재작성
- Phase 3: 프론트 아코디언 UI 연결
- Phase 4: GitHub Actions 자동화
- Phase 5: Google News RSS 보강 + 중복 제거 고도화

## 11) 비용 관점(기획안 기준)
- Naver API: 무료 구간 사용
- Google News RSS: 무료
- AI 재작성: 월 소액(건당 저비용 가정)
- GitHub Actions: public repo 무료 구간 활용
