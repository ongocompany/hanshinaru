# 한 줄 결론

AI Hub `603` **고서 한자 인식 OCR 데이터**는 한국 한시 파이프라인에 넣을 가치가 충분하다.  
다만 **직접 수집 소스**가 아니라, `PDF/이미지 -> OCR -> 원문 텍스트`를 여는 **OCR 보조 트랙**으로 넣는 것이 맞다.

# 왜 활용 가치가 큰가

AI Hub 페이지 기준으로 이 데이터셋은 아래 특성을 가진다.

- 데이터명: `고서 한자 인식 OCR 데이터`
- 성격: 조선시대 고서 원문 이미지 + 낱자 바운딩박스 + 유니코드 라벨 JSON
- 규모:
  - 한자 글자수 기준 `10,439,251자`
  - 이미지 `55,685면`
- 서체:
  - 해서 `6,841,923자`
  - 행서 `3,024,903자`
  - 초서 `529,010자`
  - 전서/예서 포함
- 활용 서비스 예시:
  - 고문헌 한자 인식(OCR)
  - 고문헌 이미지 검색
  - 고문헌 자동번역
- 제공 부가물:
  - 세그멘테이션 모델 설명
  - 클러스터 모델 설명
  - OCR 인식 모델 설명
  - 데이터 설명서 / 활용가이드 / API 다운로드

즉, 이 데이터는 “한시 작품 그 자체”라기보다,  
우리가 확보한 `고문헌 PDF/원문 이미지`를 **읽을 수 있게 만드는 학습·평가 인프라**에 가깝다.

# 현재 파이프라인과의 맞물림

현재 한국 한시 파이프라인은 대략 이렇게 가고 있다.

1. 작가/작품 후보를 보드로 고정
2. 실제 텍스트가 바로 열리는 페이지를 우선 수집
3. 텍스트가 안 열리면 문집/판본/소장처를 찾음
4. 그다음 단계가 사실상 `이미지/PDF 기반 복원`인데, 여기가 비어 있었다

AI Hub 603은 바로 **4번을 채우는 트랙**이다.

즉, 아래처럼 붙이는 게 적절하다.

```text
보드 후보 선정
-> 직접 텍스트 수집
-> 실패/부분성공 후보를 OCR 큐로 보냄
-> KORCIS/규장각/장서각/ITKC 이미지·PDF 확보
-> AI Hub 603 기반 OCR 모델 bootstrap/fine-tune
-> OCR 결과 검수
-> poemZh/locator/source policy 반영
```

# 직접 활용 시 주의점

## 1. 이 데이터는 “학습용”이지 “완성 OCR 서비스”가 아니다

- 페이지 설명상 이미지와 JSON 라벨, 그리고 모델 설명/다운로드가 중심이다.
- 따라서 바로 텍스트를 API로 뽑는 방식이 아니라:
  - 모델 bootstrap
  - fine-tune
  - 평가셋
  - 문자 분리/인식 기준 정립
  로 써야 한다.

## 2. 우리 작품 코퍼스와 1:1 대응하지 않는다

- 조선시대 고서 전반을 대상으로 만든 데이터라서,
- 우리가 지금 찾는 특정 작품(예: 정지상 `新雪`, `西樓`)의 원문이 바로 있는 것은 아니다.
- 대신 같은 종류의 활자·목판·필사본 한문 페이지를 읽는 능력을 올리는 데 유효하다.

## 3. 현재 파일럿 병목과 정확히 맞닿아 있다

지금 실제 병목은 아래다.

- 정지상: `新雪`, `鄕宴致語`, `栢律寺`, `西樓` 직접 텍스트 부재
- 최치원: `江南女`, `鄕樂雜詠` 직접 텍스트 부재
- 허난설헌: 보드 후보와 실제 접근 가능한 텍스트 목록 불일치

이건 전형적인 OCR 투입 구간이다.

# 이용 조건 확인

AI Hub 데이터 이용정책 페이지 기준:

- AI Hub 데이터는 **영리적·비영리적 연구·개발 목적**으로 활용 가능하다고 명시돼 있다.
- 다만 데이터셋 다운로드는:
  - 신청/승인 절차가 있고
  - API 다운로드는 승인 후 사용 가능
  - 분할 압축 병합 절차를 권장

즉, **활용 자체는 가능**하지만,
파이프라인 자동화에 넣으려면 먼저:

1. 형 계정으로 다운로드 승인
2. jinas 또는 jinserver에 저장
3. 압축 병합
4. 학습/추론 런타임 결정

이 선행돼야 한다.

# 파이프라인 포함 방식

이번 세션에서 아래처럼 반영하기로 판단했다.

## 포함 원칙

- `AI Hub 603`은 `poem source`가 아니다.
- `OCR bootstrap dataset`으로 취급한다.
- 실제 작품 출처는 여전히 `KORCIS / 규장각 / 장서각 / ITKC / 위키문헌 / 백과`가 맡는다.

## 실제 운영 위치

- `text-collection pilot`에서 직접 본문이 없는 후보를 `OCR queue`로 보낸다.
- OCR queue는 이미지/PDF 기반 복원 대상 목록이다.
- OCR 결과는 반드시:
  - 원문 이미지 대조
  - 권차/수록 위치 확인
  - source policy 검토
  를 거친 뒤 poem record로 승격한다.

# 지금 단계에서의 판단

- **포함해야 한다**: 예
- **지금 즉시 다운로드/학습까지 들어가야 하냐**: 아직 아님
- **왜 아직 아니냐**:
  - 현재는 후보/텍스트/권리 구조를 먼저 안정화하는 단계
  - AI Hub 603은 학습/평가 트랙이라 저장·실행 환경 결정이 먼저 필요

따라서 권장 순서는 이렇다.

1. `OCR queue` 기준으로 어떤 작품이 OCR 대상인지 고정
2. AI Hub 603 승인/다운로드
3. 저장 위치를 `jinas` 또는 `jinserver`로 결정
4. 작은 OCR 실험 1회
   - 예: `정지상` 1작품 + `최치원` 1작품
5. 성공하면 OCR 트랙을 정식 ingest 전단계로 편입
