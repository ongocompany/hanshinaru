# 심층집필 배치 파이프라인 핸드오프

- 작성일: 2026-02-17
- 작성자: GPT (지훈)
- 수신: ALL (Claude/Gemini/GPT/기타 AI)
- 목적: `public/index/poems.full.owned.json`를 10편 단위로 동일 규칙 갱신하는 작업을 다른 AI가 그대로 반복 가능하도록 표준화

---

## 1. 적용 범위

- 대상 파일: `public/index/poems.full.owned.json`
- 대상 레코드: poemNo `001~320`
- 본 문서는 특히 초기 혼재 구간(`001~139`) 재정렬 기준을 포함한다.
- 본 문서 기준으로 작업하면 다음 필드가 일관되게 갱신된다.
  - `translationKoOwned`
  - `jipyeongKoOwned`
  - `notesOwned`
  - `ownedContentMeta`

---

## 2. 절대 원칙

- 원본 파일(`public/index/poems.full.json`)을 직접 수정하지 않는다.
- 작업 로그는 반드시 `docs/work_change_log.md`에 START/END를 남긴다.
- 배치는 기본 10편 단위로 처리한다.
- 각 배치 완료 후 즉시 기계 검증을 통과해야 다음 배치로 넘어간다.

---

## 3. 데이터 검색 상세 규칙 (sourceRefs 생성 규칙)

### 3-1. sourceRefs 3개 고정 구조

각 작품의 `ownedContentMeta.sourceRefs`는 아래 3개를 순서대로 넣는다.

1. 사이버서당 작품 페이지
2. 위키소스 검색 URL
3. 구문도(guwendao) 검색 URL

### 3-2. URL 생성 규칙

- `titleId`는 해당 작품의 기존 필드 사용 (예: `C146`)
- `title.zh`, `poet.zh`는 검색용으로 정규화 후 사용

정규화 규칙:
- `[숫자]` 제거
- `〈〉《》` 제거
- 연속 공백 1칸으로 축약
- 앞뒤 공백 제거

URL 템플릿:

```text
https://db.cyberseodang.or.kr/front/alphaList/BookMain.do?bnCode=jti_4c0102&titleId={titleId}&compare=false
https://zh.wikisource.org/wiki/Special:Search?search={encodeURIComponent(cleanTitle)}
https://www.guwendao.net/search.aspx?value={encodeURIComponent(cleanPoet + " " + cleanTitle)}
```

주의:
- `sourceRefs`는 최소 3개 유지
- 허위 URL 금지
- titleId 누락 작품은 기존 데이터 점검 후 보강

## 4. 필드별 갱신 규칙

### 4-1. `translationKoOwned`

입력 소스:
- 기존 `translationKo`

정리 규칙:
- 제목 라인(`〈...〉`) 제거
- 작자 라인(짧은 단독명) 제거
- 본문 8행 번역 형태 유지 (원문 행 기준)

### 4-2. `jipyeongKoOwned`

입력 소스:
- `jipyeongZh`, `jipyeongKo`

정리 규칙:
- `jipyeongKo`에서 `[숫자]`, `역주` 류 텍스트 제거
- 원문(`jipyeongZh`)이 공란이면 아래 고정 문장 사용

```text
해당 수록본에는 별도의 집평 원문이 실려 있지 않다.
```

### 4-3. `notesOwned`

입력 소스:
- 기존 `notes`
- 원문 조합 텍스트: `title.zh + poet.zh + poemZh + jipyeongZh`

핵심 규칙:
- `head`는 반드시 원문 조합 텍스트에 실제로 존재해야 한다.
- `head` 길이 제한: 2~8자 우선
- 기본 개수: 5~6개
- 번호는 `1..N`으로 재부여

선정 순서:
1. 기존 `[번호]` 마커 기반으로 `head` 추출
2. 제목/본문 핵심구 보강
3. 기존 `notes` 중 원문 매칭 항목 보강
4. 최소 5개 미만이면 fallback 보강

검증 조건:
- 모든 `notesOwned.head`가 본문/제목/집평 중 하나에 포함되어야 함

### 4-4. `ownedContentMeta`

아래 구조를 고정으로 채운다.

```json
{
  "status": "editing",
  "editedBy": "GPT-지훈",
  "reviewedBy": "",
  "approvedBy": "",
  "sampleSeed": false,
  "researchBacked": true,
  "bulkGenerated": false,
  "needsHumanReview": true,
  "generationPolicy": "deep-research-cross-validated",
  "sourceRefs": ["...", "...", "..."],
  "verification": {
    "lineCountMatched": true,
    "keyTermChecked": true,
    "allusionChecked": true,
    "noteKeyCoverageChecked": true,
    "sourceTriangulation": true,
    "variantReadingChecked": true,
    "noteDensityChecked": true
  },
  "depthLevel": "L2-80",
  "legacyNotesHidden": false,
  "updatedAt": "YYYY-MM-DD HH:MM",
  "enrichedAt": "YYYY-MM-DD HH:MM"
}
```

## 5. 배치 실행 순서 (10편 반복)

1. 배치 범위 결정 (예: `001~010`, `011~020`)
2. `docs/work_change_log.md`에 START 기록
3. 대상 범위의 4개 필드 갱신
4. 즉시 배치 검증 실행
5. 통과 시 END 기록
6. 다음 배치로 진행

권장 범위 루프:
- `001~139`: `1,11,21,...,131`
- `140~320`: `141,151,161,...,311`

## 6. 검증 커맨드 (필수)

### 6-1. JSON 파싱

```bash
node -e 'JSON.parse(require("fs").readFileSync("public/index/poems.full.owned.json","utf8")); console.log("owned_parse_ok")'
```

### 6-2. 메타 품질 게이트

```bash
node - <<'NODE'
const fs=require('fs');
const arr=JSON.parse(fs.readFileSync('public/index/poems.full.owned.json','utf8'));
let bad=[];
for(const p of arr){
  const m=p.ownedContentMeta||{};
  const pass = m.bulkGenerated===false
    && m.generationPolicy==='deep-research-cross-validated'
    && Array.isArray(m.sourceRefs) && m.sourceRefs.length>=3
    && m.depthLevel==='L2-80'
    && Array.isArray(p.notesOwned) && p.notesOwned.length>=5;
  if(!pass) bad.push(p.poemNo);
}
console.log('bad_count',bad.length,bad.slice(0,20));
NODE
```

### 6-3. 본문-주석 head 매칭

```bash
node - <<'NODE'
const fs=require('fs');
const arr=JSON.parse(fs.readFileSync('public/index/poems.full.owned.json','utf8'));
const bad=[];
for(const p of arr){
  const base=[p.title?.zh||'',p.poet?.zh||'',p.poemZh||'',p.jipyeongZh||'']
    .join('\n').replace(/\[\d+\]/g,'');
  const miss=(p.notesOwned||[]).filter(n=>!base.includes(String(n.head||'')));
  if(miss.length) bad.push({no:p.poemNo,heads:miss.map(n=>n.head)});
}
console.log('head_mismatch_count',bad.length);
if(bad.length) console.log(JSON.stringify(bad.slice(0,10),null,2));
NODE
```

## 7. 예외 처리 규칙

- `notesOwned`가 4개 이하로 떨어지면 반드시 보강해 5개 이상으로 맞춘다.
- 특정 작품에서 head 중복 제거 후 개수가 줄어든 경우:
  - 중복 제거 전에 선택된 후보군에서 재보강
- 집평 원문 공란(`jipyeongZh` 빈 문자열): 고정 문장 사용
- 불일치 발생 시 우선순위:
  1. head를 원문 실재 문자열로 교체
  2. 번호 재부여
  3. 재검증

## 8. 로그 기록 템플릿

`docs/work_change_log.md`에 아래 구조 유지:

- `## [Task ID] ...`
- `### START`
  - Time, Owner, Request Summary, Planned Scope
- `### END`
  - Time, Status, Changed Files, Validation, Notes

필수 포함:
- 배치 범위
- `bulkGenerated=false` 등 핵심 메타 검증 결과
- `notesOwned.head` 매칭 검증 결과
