## 1. 개요

당시삼백수 프로젝트의 JSON 데이터를 웹 브라우저에서 조회/수정/검증할 수 있는 관리툴.
별도 서버 없이 **정적 HTML 파일**로 제작하며, JSON 파일을 불러와 편집 후 다운로드하는 방식.

---

## 2. 관리 대상 데이터

### 2-1. 3개 JSON 파일

| 파일 | 레코드 수 | 핵심 역할 |
|------|----------|----------|
| `db_author.with_ko.json` | 76명 | 시인 정보 + 출생지 + 교유관계 |
| `poems.full.json` | 320편 | 시 본문 + 번역 + 주석 |
| `history_cards.json` | 36건 | 역사 사건 카드 |

### 2-2. 파일 간 연결 구조

```
시인(author)                  시(poem)                     역사(history)
─────────────               ─────────────               ─────────────
titleId: "C382"             poet.zh: "張九齡[1]"         titleId: "H001"
name.zh: "張九齡"     ←──── 정규화 매핑 ────→            year: 618
era: "early"                category: "五言古詩"          tags.era: ["초당"]
relations: [...]            notes: [...]                  annotations: [...]
birthplace: {...}           poemNo: 1
```

**핵심 매핑**: poem의 `poet.zh`에서 주석번호(`[1]`)를 제거 → author의 `name.zh`와 매칭

### 4-4. 데이터 검증 (Validator)

전체 데이터 정합성을 한 번에 검사하는 기능.

| 검증 항목 | 설명 | 심각도 |
|----------|------|--------|
| 시인↔시 매핑 | poem.poet.zh 정규화 → author.name.zh 일치 여부 | ERROR |
| 필수 필드 누락 | titleId, name, era 등 필수값 비어있는지 | ERROR |
| era 일관성 | author.era와 birth year 범위 불일치 | WARN |
| 관계도 양방향 | A→B 있으면 B→A도 있는지 | WARN |
| 외부인물 필드 | EXT_ 관계에 targetName/targetNameKo 있는지 | ERROR |
| 주석 번호 정합성 | 본문의 [N]과 notes.no 매핑 누락 | WARN |
| titleId 중복 | 전체 데이터에서 ID 중복 여부 | ERROR |
| 좌표 범위 | lat -90~90, lng -180~180 범위 내인지 | WARN |

#### 검증 결과 화면
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  데이터 검증 결과
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  ✅ 통과: 12개 항목
  ⚠️  경고: 3개 항목
  ❌ 오류: 1개 항목

  ❌ [시인↔시 매핑] C331 皎然 - 매칭되는 시 없음
  ⚠️  [관계도 양방향] C328→C401 있으나 C401→C328 없음
  ⚠️  [출생지 누락] 72명 birthplace 미입력
```

## 5. 기술 구현 방식

### 5-1. 아키텍처

```
admin.html (단일 파일 또는 admin/ 폴더)
  ├── JSON 파일 로드 (FileReader API 또는 fetch)
  ├── 메모리에서 편집 (JavaScript 객체 조작)
  ├── 변경사항 미리보기
  └── JSON 다운로드 (Blob + download)
```

- **서버 불필요**: 브라우저에서 파일 열기 → 편집 → 다운로드
- **라이브러리 최소화**: 기존 프로젝트와 동일하게 Vanilla JS 기본
- **Leaflet**: 출생지 좌표 미리보기용 (기존 CDN 재사용)

### 5-2. 데이터 흐름

```
[파일 선택] → FileReader로 JSON 파싱
     ↓
[메모리 상태] → 화면에 렌더링 (목록/상세)
     ↓
[사용자 편집] → 메모리 상태 업데이트
     ↓
[검증 실행] → 오류/경고 표시
     ↓
[저장(다운로드)] → JSON.stringify → Blob → 다운로드
     ↓
[수동 교체] → 다운로드된 JSON을 public/index/에 덮어쓰기
```

### 5-3. 파일 구조 (안)

```
admin/
├── index.html          # 관리툴 메인 (탭 네비게이션)
├── admin.css           # 관리툴 전용 스타일
├── admin.js            # 핵심 로직 (데이터 로드/저장/검증)
├── author-manager.js   # 시인 CRUD
├── poem-manager.js     # 시 CRUD
├── history-manager.js  # 역사 CRUD
└── validator.js        # 데이터 검증
```

## 6. 작업 순서 (우선순위)

| 단계 | 작업 | 의존성 |
|------|------|--------|
| 1단계 | admin.html 기본 틀 + 탭 네비게이션 + JSON 로드/저장 | 없음 |
| 2단계 | 시인 관리 (목록 + 상세편집) | 1단계 |
| 3단계 | 시인 출생지/관계 편집 (Leaflet 미리보기 포함) | 2단계 |
| 4단계 | 데이터 검증 (Validator) | 2단계 |
| 5단계 | 시 관리 (목록 + 상세편집 + 주석편집) | 1단계 |
| 6단계 | 역사 관리 (목록 + 상세편집 + annotation편집) | 1단계 |

### 1단계 완료 시 가능한 것
- JSON 파일 불러오기/내려받기
- 탭으로 시인/시/역사/검증 메뉴 전환

### 2~3단계 완료 시 가능한 것
- 시인 정보 CRUD (특히 태훈이가 보낸 birthplace/relations 데이터 입력/수정)
- 지도에서 좌표 확인

### 4단계 완료 시 가능한 것
- 데이터 입력 후 정합성 한 번에 검증
- 누락/불일치 항목 바로 확인

## 7. 참고: poems.compact.json 처리

`poems.compact.json`은 `poems.full.json`의 서브셋이므로 별도 관리 불필요.
시 관리에서 poems.full.json을 수정하면, **저장 시 compact 버전도 자동 생성**하는 기능 포함.

compact에 포함되는 필드: `titleId`, `poemNo`, `poemNoStr`, `title`, `poet`, `juan`, `category`, `meter`, `preview`
