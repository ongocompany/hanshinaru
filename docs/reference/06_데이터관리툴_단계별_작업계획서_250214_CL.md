# 1단계: 기본 틀 + JSON 로드/저장

> 목표: admin/ 폴더에 관리툴 껍데기를 만들고, JSON을 불러와서 저장하는 파이프라인 완성

## 사전 결정 사항 [형] 

| # | 결정 항목 | 선택지 | 추천 | 이유 |
|---|----------|--------|------|------|
| Q1 | 관리툴 위치 | (A) `admin/` 폴더 (B) 루트에 `admin.html` 1개 | A | 파일이 6개 이상이라 폴더로 분리가 깔끔 |
| Q2 | JSON 로드 방식 | (A) `public/index/`에서 fetch 자동 로드 (B) 파일 선택(FileReader)으로 수동 로드 (C) 둘 다 | C | 개발 중엔 자동 로드, 배포 후엔 수동 로드 필요 | jin-개발후에도 굳이 수동로드가 필요 없을듯. 
| Q3 | 저장 방식 | (A) JSON 다운로드만 (B) 다운로드 + `public/index/`에 직접 fetch로 덮어쓰기 | A | 서버 없이는 직접 덮어쓰기 불가, 다운로드 후 수동 교체 | jin-직접엎어 쓰는 방식이 아예 없음? 엄청 불편할거 같은데. 
| Q4 | CSS 스타일 | (A) 메인 사이트와 동일 톤 (B) 완전 별개 (심플한 관리자 UI) | B | 관리툴은 기능 중심, 꾸미기 최소화 | jin-B, 단 가독성은 좋아야 함. 입력시 실수 방지 대책 세울것

## 작업 목록

### 1-1. 폴더 및 파일 생성 [AI]
```
admin/
├── index.html       # 메인 페이지 (탭 네비게이션)
├── admin.css        # 관리툴 스타일
└── admin.js         # 데이터 로드/저장 + 탭 전환 + 공용 유틸
```

**산출물**: 브라우저에서 `admin/index.html` 열면 빈 탭 4개가 보이는 상태

### 1-2. JSON 로드 기능 [AI]
- `public/index/` 경로에서 3개 JSON fetch
- 로드 성공/실패 상태 표시 (초록/빨강 인디케이터)
- 파일별 레코드 수 표시 (시인 76명, 시 320편, 역사 36건)
- FileReader로 수동 업로드 버튼도 제공

### 1-3. JSON 저장(다운로드) 기능 [AI]
- "저장" 버튼 클릭 시 수정된 JSON을 Blob으로 다운로드
- 파일명에 타임스탬프 포함: `db_author.with_ko_250214_1430.json`
- 변경사항 있으면 탭 제목에 `*` 표시 (예: `시인관리 *`)
- 저장 안 하고 탭 이동 시 경고

### 1-4. 탭 네비게이션 [AI]
- 4개 탭: 시인관리 / 시관리 / 역사관리 / 데이터검증
- 각 탭 클릭 시 해당 패널만 표시
- URL hash로 탭 상태 유지 (`admin/#author`, `admin/#poem` 등)

## 완료 기준
- [ ] `admin/index.html`을 브라우저에서 열 수 있음
- [ ] 3개 JSON 파일이 자동 로드되고 레코드 수가 표시됨
- [ ] 수동 파일 업로드로도 JSON 로드 가능
- [ ] JSON 다운로드가 정상 동작 (내용 보존 확인)
- [ ] 4개 탭 전환이 동작함

## [형] 확인 포인트

1단계 끝나면 형이 확인할 것:

# 2단계: 시인 관리 — 목록 + 상세 편집

> 목표: 76명 시인의 기본 정보를 목록에서 보고, 선택해서 편집할 수 있는 화면

## 사전 결정 사항 [형]

| # | 결정 항목 | 선택지 | 추천 | 이유 |
|---|----------|--------|------|------|
| Q5 | 목록 기본 정렬 | (A) titleId순 (B) 이름 가나다순 (C) 출생연순 | C | 타임라인 프로젝트니까 시간순이 자연스러움 | JIn-A는 제외하고 드롭다운메뉴로 가나다순, 연도순, 작품등재순 (작품 많은 순서부터위로, 작품수 동일시 가나다순)
| Q6 | 편집 화면 위치 | (A) 목록 아래에 펼침 (B) 목록 옆에 나란히 (좌=목록, 우=편집) (C) 모달로 열기 | B | 목록 보면서 편집 가능, 가장 효율적 | Jin-B
| Q7 | 시인 새로 추가 기능 | (A) 필요함 (B) 불필요 (기존 76명만 편집) | B | 당시삼백수 시인은 고정, 필드 수정만 필요 | Jin-A , 단 (당시삼백수 미등재 시인)이라는 필드 추가 필요.

## 작업 목록

### 2-1. 시인 목록 테이블 [AI]

| 컬럼 | 데이터 | 너비 |
|------|--------|------|
| ID | titleId | 60px |
| 이름 | name.ko (name.zh) | 150px |
| 생몰년 | life.raw | 100px |
| 시대 | era.period 배지 | 70px |
| 작품수 | poems.full에서 매칭 카운트 | 50px |
| 출생지 | birthplace 유무 (O/X) | 50px |
| 관계 | relations 개수 | 50px |

- 컬럼 헤더 클릭 시 정렬 토글 (오름/내림)
- 행 클릭 시 우측 편집 패널에 상세 로드
- 현재 선택된 행 하이라이트

### 2-2. 검색 + 필터 바 [AI]
- 텍스트 검색: 이름(한/중) 실시간 필터
- era 필터: 전체 / early / high / mid / late 버튼 그룹
- 상태 필터: 전체 / 출생지 있음 / 출생지 없음 / 관계 있음 / 관계 없음

### 2-3. 시인 상세 편집 폼 [AI]

**기본정보 섹션**:
- titleId (읽기전용, 회색 배경)
- name.zh / name.ko (텍스트 입력)
- life.birth / life.death (숫자 입력)
- life.birthApprox / life.deathApprox (체크박스)
- life.raw (텍스트 입력)

**시대 분류 섹션**:
- era.period: 라디오 버튼 4개 (early/high/mid/late)
- era.confidence: 드롭다운 (high/medium/low)
- era.source: 드롭다운 (birth_year/bio_era/activity/manual)

**약전 섹션**:
- bioKo: textarea (높이 자동 조절)

**작품 목록 섹션 (읽기전용)**:
- 이 시인과 매핑되는 시 목록 표시 (poemNo + 제목)
- 매핑 기준: normalizePoetName(poem.poet.zh) === author.name.zh
- 매핑 실패 시 빨간색 경고 표시

### 2-4. 편집 → 저장 파이프라인 [AI]
- 폼 값 변경 시 메모리 내 JSON 객체 즉시 업데이트
- 변경 감지: 원본 vs 현재 상태 비교 → 변경된 필드 하이라이트
- "이 시인 되돌리기" 버튼: 해당 시인만 원본으로 복원
- "전체 저장" 버튼: 변경된 JSON 다운로드

### 2-5. 변경 추적 표시 [AI]
- 목록에서 수정된 시인 행에 노란 배경 표시
- 상단에 "N건 수정됨" 카운터 표시
- 수정 내역 요약 팝업 (어떤 시인의 어떤 필드가 변경되었는지)

## 완료 기준
- [ ] 76명 시인이 테이블로 표시됨
- [ ] 검색/필터/정렬이 모두 동작함
- [ ] 시인 선택 → 상세 편집 → 값 변경 → 목록에 반영됨
- [ ] 변경 추적이 동작함 (수정된 행 하이라이트, 카운터)

# 4단계: 데이터 검증기

## 작업 목록

### 4-1. 검증 규칙 구현 [AI]

**ERROR (반드시 수정해야 함)**:

| 규칙 | 검증 내용 | 수정 가이드 |
|------|----------|-----------|
| E01 | titleId 중복 | 중복 ID 목록 표시 |
| E02 | 필수 필드 누락 (name.zh, name.ko, era.period) | 누락 필드 표시 |
| E03 | poem.poet.zh → author.name.zh 매핑 실패 | 어떤 시가 매핑 안 되는지 표시 |
| E04 | EXT_ 관계에 targetName/targetNameKo 누락 | 해당 relation 표시 |
| E05 | life.birth > life.death (출생이 사망보다 나중) | 해당 시인 표시 |

**WARNING (확인 권장)**:

| 규칙 | 검증 내용 | 수정 가이드 |
|------|----------|-----------|
| W01 | birthplace 누락 | 미입력 시인 수 + 목록 |
| W02 | relations 빈 배열 | 관계 없는 시인 수 + 목록 |

# 단계 간 의존성 & 예상 작업량

```
1단계 (기본 틀)          ← 모든 단계의 기반
  ↓
2단계 (시인 기본)        ← 가장 중요, 태훈이 데이터 입력에 필수
  ↓
3단계 (출생지/관계)      ← 2단계에 의존, 태훈이 데이터 머지 기능 포함
  ↓
4단계 (검증기)           ← 2단계에 의존, 데이터 품질 보장

5단계 (시 관리)          ← 1단계에만 의존, 독립적
6단계 (역사 관리)        ← 1단계에만 의존, 독립적
```

**권장 진행 순서**: 1 → 2 → 3 → 4 → 5 → 6
**이유**: 태훈이 데이터가 들어오면 3단계 임포트 기능으로 바로 입력 가능
